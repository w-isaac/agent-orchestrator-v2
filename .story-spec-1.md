# AOV-1: Set up TypeScript/Express server with Docker Compose, health endpoint, and migration runner

As a developer onboarding to the v2 project, I need a fully scaffolded TypeScript/Express server with Docker Compose (app + Postgres with pgvector), a working health endpoint, a DB connection pool, and a migration runner so that the foundational infrastructure is in place for all subsequent backend development.

This story establishes the canonical project structure (src/routes, src/services, src/adapters, src/lib), ensuring every team member and agent starts from a consistent, runnable baseline. Without this foundation, no other backend stories can be started.


## Acceptance Criteria
- Given the repo is cloned, when `docker compose up` is run, then the app container and a Postgres+pgvector container start without errors
- Given the stack is running, when GET /api/health is called, then it returns HTTP 200 with a JSON body indicating service status and DB connectivity
- Given the stack is running, then a connection pool to Postgres is initialised on app startup and shared across the app
- Given migrations exist in a designated folder, when the app starts (or a migration command is run), then all pending migrations are applied in order and the result is logged
- Given a fresh clone, when the developer copies .env.example to .env and runs docker compose up, then the app is fully operational with no manual DB setup
- Given the project structure, then src/routes, src/services, src/adapters, and src/lib directories exist with index barrel files so future stories can add modules without restructuring

## Technical Approach

Bootstrap a TypeScript/Express server inside Docker Compose with two services: `app` (Node.js) and `db` (Postgres 16 + pgvector). The app follows a layered directory convention (`routes`, `services`, `adapters`, `lib`) with barrel exports. Startup sequence is sequential and deterministic: **init pool → run migrations → start HTTP server**. A standalone migration runner reads `.sql` files from a designated directory, tracks applied migrations in a `schema_migrations` table, and applies pending ones in filename-sort order. The health endpoint probes the pool and returns a structured JSON contract suitable for load balancers and monitoring.

**Key design decisions:**
- **Exit on migration failure** (safer than degraded-start; Docker restart policy handles retries; logs remain inspectable via `docker compose logs`)
- **Migration table named `schema_migrations`** (avoids collision with app-domain "migrations" concept; aligns with Rails/Knex conventions)
- **`/api/health` unauthenticated** (standard for LB probes; no sensitive DB metadata exposed)
- **`pg` driver directly** (no ORM — keeps the scaffold minimal; future stories can layer Drizzle/Kysely on top)
- **`ts-node-dev`** for local dev hot-reload; production builds compile to JS

## Risk Assessment

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| 1 | **DB not ready when app starts** — Postgres takes longer to init than Node | High (first `docker compose up`) | App crashes on pool connect / migration | Use `depends_on` with `service_healthy` + pg healthcheck in compose; retry loop in `createPool()` with 5 attempts, 2s backoff |
| 2 | **Migration ordering ambiguity** — non-numeric filenames sort unexpectedly | Low | Wrong migration order, schema corruption | Enforce `NNN_name.sql` naming convention; sort filenames lexicographically; document convention in README |
| 3 | **Partial migration failure** — SQL error mid-file leaves schema in broken state | Medium | Stuck state requiring manual DB fix | Wrap each migration file in a transaction (`BEGIN`/`COMMIT`); on error, `ROLLBACK` and exit non-zero |
| 4 | **`pgvector` extension not available** — wrong base image used | Low | `CREATE EXTENSION vector` fails | Pin Docker image to `pgvector/pgvector:pg16`; test in CI |
| 5 | **Port conflict** — developer already has something on `:3000` or `:5432` | Medium | `docker compose up` fails with bind error | `.env.example` documents ports; compose maps from env vars so developer can override |
