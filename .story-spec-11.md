# AOV-11: Codex adapter: full prompt with embedded context, async polling, and result normalization

As an agent orchestrator, I need the CodexAdapter to implement the AgentAdapter interface fully — embedding context directly into the prompt (no tool calls), submitting tasks asynchronously to the OpenAI API, polling for completion, normalizing results into the shared adapter contract, and handling errors with retry logic — so that Codex can participate as a dispatched agent alongside Claude Code and Gemini.


## Acceptance Criteria
- Given a task with context, when CodexAdapter.dispatch() is called, then the prompt is constructed by embedding all context inline (no tool/function calls used)
- When the prompt is submitted to OpenAI API, then the request is made asynchronously and a run/job ID is returned without blocking
- Given a pending job ID, when the adapter polls for completion, then it retries at a configured interval until the run reaches a terminal state (completed, failed, or cancelled)
- When the run completes successfully, then the raw OpenAI response is normalized into the standard AgentAdapter result shape (output, token usage, metadata)
- When the API returns a transient error (5xx, rate limit), then the adapter retries up to a configured maximum with exponential backoff before surfacing the error
- Given a failed or cancelled run, then the adapter returns a structured error result that the orchestrator can use for conflict resolution or re-queue logic

## Technical Approach

Implement a `CodexAdapter` that mirrors the existing agent pattern (async function, run record creation, artifact persistence, WebSocket broadcasting) but targets the **OpenAI Responses API** instead of the Claude CLI. Key differences from existing agents:

1. **Embedded context prompt** — All story context, acceptance criteria, and upstream artifacts are concatenated inline into a single prompt string. No tool/function calls.
2. **Async submission + polling** — Submit via `POST /v1/responses` (or Codex-specific endpoint), receive a run/job ID, then poll at a configurable interval until terminal state.
3. **Result normalization** — Map OpenAI's response shape (`output`, `usage.input_tokens`, `usage.output_tokens`, `model`, `status`) into the shared `agent_runs` contract.
4. **Retry with exponential backoff** — Transient errors (5xx, 429) trigger retry with backoff; terminal failures surface structured error results.

The adapter plugs into the existing scheduler via `getAgentRunner()` — no scheduler changes needed beyond registering the new mapping.

## Data Model Changes

### New Table: `codex_poll_attempts`

Tracks each polling attempt and retry for audit/UI display in the `RetryErrorAlert` component.

```sql
CREATE TABLE IF NOT EXISTS codex_poll_attempts (
  id          TEXT PRIMARY KEY,
  run_id      TEXT NOT NULL REFERENCES agent_runs(id),
  attempt     INTEGER NOT NULL,
  status      TEXT NOT NULL,          -- 'pending', 'polling', 'success', 'error'
  http_status INTEGER,                -- HTTP status code from OpenAI
  error_type  TEXT,                   -- 'rate_limit', '5xx', 'timeout', 'cancelled', null
  error_msg   TEXT,
  backoff_ms  INTEGER,                -- backoff delay applied before this attempt
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_codex_poll_run ON codex_poll_attempts(run_id);
```

### Altered Table: `agent_runs`

Add Codex-specific columns:

```sql
ALTER TABLE agent_runs ADD COLUMN openai_run_id TEXT;          -- OpenAI job/run ID for correlation
```

## Risk Assessment

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **OpenAI rate limits (429)** — High-volume dispatches could exhaust rate limits, stalling all Codex runs | Medium | Exponential backoff with jitter; configurable max retries (default 5); circuit breaker integration via existing `consecutiveFailures` in scheduler |
| 2 | **Polling cost** — Frequent polling wastes resources; infrequent polling adds latency | Low | Configurable interval (default 3s) with adaptive backoff: short initial polls, lengthening over time (3s → 5s → 10s) |
| 3 | **Prompt size limits** — Embedding all context inline could exceed OpenAI model context window | Medium | Calculate token estimate before dispatch; if exceeding threshold (~90% of model context), truncate oldest context sections with a warning logged to `agent_events` |
| 4 | **Stale polling** — Network partition or process restart loses in-flight poll state | Medium | Persist `openai_run_id` on `agent_runs` immediately after dispatch; on scheduler restart, query for runs with `adapter_type='codex'` and `status='running'` and resume polling |
| 5 | **Backward compatibility** — New `agent_runs` columns must not break existing Claude/Gemini queries | Low | All new columns are nullable with sensible defaults; existing `SELECT *` queries gain columns transparently; `adapter_type` defaults to `'claude'` for existing rows |
| 6 | **API key management** — OpenAI API key must be securely stored and not leaked in logs | High | Read from `OPENAI_API_KEY` env var only; never log or persist the key; use the SDK's built-in auth (reads env var automatically) |
