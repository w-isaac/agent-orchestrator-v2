# AOV-12: Gemini adapter: full prompt with embedded context, async polling, and result normalization

As an orchestrator, I need a complete GeminiAdapter that implements the AgentAdapter interface so that research and synthesis tasks with large context windows (>100K tokens, up to 1M+) can be routed to and executed by Google Gemini models. The adapter must embed the full context directly in the prompt payload, submit to the Google AI API, handle async polling for completion, and normalize results back into the orchestrator's standard format. This unlocks Gemini's uniquely large context window as a routing target for tasks that exceed the capacity of other adapters.


## Acceptance Criteria
- Given GeminiAdapter is instantiated, when inspected, it fully satisfies the AgentAdapter interface contract (same as Claude/Codex adapters)
- Given a task with embedded context, when the adapter builds the prompt, then the full context (up to 1M tokens) is embedded directly in the prompt body without truncation
- Given a prompt submission, when the adapter calls the Google AI API, then it uses the correct REST endpoint, authentication, and model parameters, and returns a job/operation reference
- Given a submitted job reference, when the adapter polls for completion, then it uses exponential backoff, handles transient errors, and resolves when the generation is complete
- Given a completed Gemini response, when the adapter normalizes results, then the output conforms to the standard AgentResult schema used by the task lifecycle (story #9)
- Given a task with >100K context tokens, when the routing engine evaluates adapters, then GeminiAdapter is selected as a candidate and the full context routes correctly end-to-end

# Architecture Specification: GeminiAdapter ŌĆö Full Context, Async Polling, Result Normalization

**Story**: As an orchestrator, I need a complete GeminiAdapter that implements the AgentAdapter interface so that research and synthesis tasks with large context windows (>100K tokens, up to 1M+) can be routed to and executed by Google Gemini models.

**Date**: 2026-04-14

## Technical Approach

The orchestrator currently dispatches work via stage-specific agent functions (`runPoAgent`, `runDesignAgent`, etc.) that all invoke `runClaude()` in `src/agents/claude-runner.js`. There is no formal adapter abstraction ŌĆö adding Gemini requires introducing one.

**Strategy**:

1. **Introduce an `AgentAdapter` contract** ŌĆö a documented interface (JSDoc, not TypeScript) that both the existing Claude path and the new Gemini path satisfy. This is a lightweight shim, not a refactor of every agent.
2. **Create `gemini-runner.js`** ŌĆö analogous to `claude-runner.js`. Handles prompt construction with embedded context, submission to the Google AI `generateContent` REST endpoint, exponential-backoff polling, and result normalization to a standard `AgentResult` shape.
3. **Add a routing layer** ŌĆö a `selectAdapter()` function in the scheduler that evaluates context token count against adapter capacity thresholds. Gemini is selected when context exceeds 100K tokens.
4. **Store adapter configuration in the DB** ŌĆö a new `adapter_configs` table replaces hardcoded model settings for Gemini (API key, model, polling params). Claude config remains in `agent-models.js` for now.
5. **Extend the frontend** per the UX spec ŌĆö adapter admin panel, polling status indicator, context size banner, routing decision table, and normalized result display.

**Key constraint**: The Gemini API may return results synchronously for small prompts but requires async polling for large-context generations. The adapter must handle both paths transparently.

## Data Model Changes

> **Note**: This project uses **SQLite** (better-sqlite3), not PostgreSQL. All migrations are `.sql` files in `src/db/migrations/`.

### New Tables

#### Migration `039_gemini_adapter.sql`

```sql
-- Adapter configuration (initially for Gemini; extensible to future adapters)
CREATE TABLE IF NOT EXISTS adapter_configs (
  id            TEXT PRIMARY KEY,
  adapter_type  TEXT NOT NULL UNIQUE,            -- 'gemini', 'claude', etc.
  status        TEXT NOT NULL DEFAULT 'inactive', -- 'active' | 'inactive'
  model         TEXT NOT NULL,                    -- 'gemini-1.5-pro', 'gemini-1.5-flash'
  api_key       TEXT,                             -- encrypted at rest (app-level)
  max_context_tokens INTEGER NOT NULL DEFAULT 1048576,
  config        TEXT NOT NULL DEFAULT '{}',       -- JSON: polling params, etc.
  created_at    TEXT NOT NULL,
  updated_at    TEXT NOT NULL
);

-- Track async Gemini jobs for polling
CREATE TABLE IF NOT EXISTS gemini_jobs (
  id              TEXT PRIMARY KEY,
  run_id          TEXT NOT NULL REFERENCES agent_runs(id),
  story_id        TEXT REFERENCES stories(id),
  operation_name  TEXT,                           -- Google API operation ref
  status          TEXT NOT NULL DEFAULT 'submitted', -- submitted|polling|complete|failed
  attempts        INTEGER NOT NULL DEFAULT 0,
  last_poll_at    TEXT,
  next_poll_at    TEXT,
  backoff_ms      INTEGER NOT NULL DEFAULT 1000,
  error_code      TEXT,
  error_message   TEXT,
  raw_response    TEXT,                           -- Full Gemini API response JSON
  created_at      TEXT NOT NULL,
  completed_at    TEXT
);

CREATE INDEX idx_gemini_jobs_run_id ON gemini_jobs(run_id);
CREATE INDEX idx_gemini_jobs_status ON gemini_jobs(status);

-- Extend agent_runs to track which adapter was used
ALTER TABLE agent_runs ADD COLUMN adapter_type TEXT DEFAULT 'claude';
ALTER TABLE agent_runs ADD COLUMN gemini_model TEXT;
```

#### Migration `040_routing_decisions.sql`

```sql
-- Log routing decisions for debugging/display
CREATE TABLE IF NOT EXISTS routing_decisions (
  id              TEXT PRIMARY KEY,
  story_id        TEXT NOT NULL REFERENCES stories(id),
  run_id          TEXT REFERENCES agent_runs(id),
  context_tokens  INTEGER NOT NULL,
  evaluated       TEXT NOT NULL,                  -- JSON array of adapter evaluations
  selected_adapter TEXT NOT NULL,                 -- 'claude' | 'gemini'
  selection_reason TEXT NOT NULL,
  override        INTEGER NOT NULL DEFAULT 0,     -- 1 if manually overridden
  override_reason TEXT,
  created_at      TEXT NOT NULL
);

CREATE INDEX idx_routing_decisions_story ON routing_decisions(story_id);
CREATE INDEX idx_routing_decisions_adapter ON routing_decisions(selected_adapter);
```

## Risk Assessment

### 1. Gemini API Rate Limits & Quotas
- **Risk**: Google AI API has per-minute and per-day token quotas that vary by model. Large-context requests (>500K tokens) may hit rate limits.
- **Mitigation**: `adapter_configs.config` stores rate limit settings. The polling loop respects `429` responses with `Retry-After` headers. Surface quota errors clearly in the admin UI.

### 2. Polling Timeout for Very Large Contexts
- **Risk**: 1M-token prompts may take several minutes to generate. If the polling loop exhausts `max_retries` before completion, the task fails spuriously.
- **Mitigation**: Default `max_retries: 10` with `max_backoff_ms: 32000` yields ~5 min total polling window. Make these configurable per-adapter in the admin UI. Log polling timeline for debugging. The `gemini_jobs` table preserves state for potential manual retry.

### 3. Token Count Estimation Mismatch
- **Risk**: Client-side and server-side token counts may diverge, causing the routing engine to select the wrong adapter (e.g., estimating 95K when actual is 105K).
- **Mitigation**: Use the **same** server-side `tokenCounter.js` for both the routing decision and the UI display. The 100K threshold has buffer ŌĆö Claude's 200K capacity means only truly large contexts get misrouted. Add a `routing_decisions` audit log so mismatches can be diagnosed.

### 4. API Key Security
- **Risk**: Gemini API key stored in SQLite `adapter_configs` table is readable by anyone with DB access.
- **Mitigation**: Encrypt the API key at the application layer before storage (AES-256-GCM with a server-side secret). Never return the full key in API responses ŌĆö mask all but the last 4 characters. The admin UI uses a password field with reveal toggle.

### 5. Backward Compatibility ŌĆö `agent_runs` Schema
- **Risk**: Adding `adapter_type` and `gemini_model` columns to `agent_runs` could break existing queries that don't expect these columns.
- **Mitigation**: Both columns have defaults (`adapter_type DEFAULT 'claude'`, `gemini_model` nullable). Existing rows are unaffected. All existing queries use `SELECT *` or explicit column lists that don't include these fields, so they continue to work. The migration uses `ALTER TABLE ADD COLUMN`, which is non-destructive in SQLite.

### 6. Gemini Response Format Divergence
- **Risk**: Gemini's response structure differs from Claude's. Fields like `usage.promptTokenCount` vs `input_tokens`, `candidates[0].content.parts[0].text` vs `result` ŌĆö normalization bugs could lose data or misreport metrics.
- **Mitigation**: The `normalizeResult()` function in `gemini-adapter.js` maps Gemini fields to the standard `AgentResult` shape with explicit field-by-field mapping. Unit tests cover the mapping with real Gemini response fixtures. The `gemini_jobs.raw_response` column preserves the original response for debugging.
