# AOV-13: Agent routing: task type affinity, historical performance, and cost-aware selection

As the orchestrator, I need a routing engine that selects the best agent for each task based on task type affinity rules, historical first-try success rates, and effective cost (price × tokens / success_rate), so that tasks are dispatched to the most reliable and cost-efficient agent rather than always defaulting to claude_code. The capability matrix must be configurable, and all routing decisions must be logged for analytics and debugging.


## Acceptance Criteria
- Given a task with a known type, when the router selects an agent, then it applies the configured task type affinity rules to filter or rank eligible agents
- Given multiple eligible agents, when computing selection scores, then effective cost is calculated as price * tokens / success_rate using historical data for each agent
- Given a task type with no historical success data for an agent, when routing, then the system defaults to claude_code as the fallback agent
- Given a completed task, when the result is recorded, then the agent's first-try success rate for that task type is updated in the persistent store
- Given a routing decision is made, when any agent is selected, then the decision (task_id, agent, task_type, effective_cost, affinity_score) is written to the routing log
- Given the capability matrix configuration, when it is updated (e.g. via config file or API), then routing decisions reflect the new matrix without requiring a server restart

# Architecture Spec: Agent Routing Engine — Task Type Affinity, Historical Performance & Cost-Aware Selection

## Technical Approach

Introduce a **routing engine** between the scheduler and agent dispatch that replaces the current hard-coded `getAgentRunner()` lookup with a scoring pipeline:

1. **Filter** — capability matrix determines which agents are eligible for a task type
2. **Score** — rank eligible agents by `effective_cost = price × tokens / success_rate` combined with affinity rank
3. **Fallback** — if no agent has historical data for a task type, default to `claude_code`
4. **Log** — persist every routing decision with full scoring breakdown
5. **Update** — after task completion, update the agent's first-try success stats

The routing engine is a new service (`src/services/routingEngine.js`) called by the scheduler at dispatch time. The capability matrix is stored in the database (not a config file) so it can be updated via the UI without restart. A new set of API endpoints exposes routing config, decisions, logs, and analytics. The React frontend adds four new pages per the UX spec.

**Design decisions:**
- **Database-backed config** (not file): enables hot-reload guarantee and multi-instance safety. Resolves Open Question #1.
- **Polling** for dashboard updates (30s interval): simpler than SSE for this read-heavy, non-latency-critical view. The existing WebSocket can broadcast `routing_decision` events for instant updates if needed later. Resolves Open Question #2.
- **Fallback taxonomy**: three distinct types logged — `affinity_fallback`, `no_data_fallback`, `all_agents_failed`. Resolves Open Question #3.
- **Rolling 30-day window** for success rate calculation with an all-time fallback for agents with <10 attempts in the window. Resolves Open Question #4.
- **No auth gating** for agent performance pages — all views are operator-accessible (matches existing app pattern). Resolves Open Question #5.

## Data Model Changes

### New Tables

#### `capability_matrix`
Stores task-type-to-agent affinity rules. One row per (task_type, agent_role) pair.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | TEXT | PK | UUID |
| `task_type` | TEXT | NOT NULL | Pipeline stage / task category (e.g. `requirements`, `design`, `architecture`) |
| `agent_role` | TEXT | NOT NULL | Agent identifier (e.g. `po`, `architect`, `engineering`) |
| `affinity_rank` | INTEGER | NOT NULL DEFAULT 5 | 1 (strongest) – 10 (weakest) |
| `enabled` | INTEGER | NOT NULL DEFAULT 1 | 0 = disabled, 1 = enabled |
| `notes` | TEXT | | Operator notes |
| `created_at` | TEXT | NOT NULL | ISO timestamp |
| `updated_at` | TEXT | NOT NULL | ISO timestamp |

**Indexes:**
- `UNIQUE(task_type, agent_role)`
- `idx_capability_matrix_task_type ON capability_matrix(task_type)`

**Seed data:** Pre-populate from current `getAgentRunner()` mapping (requirements→po, design→design, etc.) with `affinity_rank=1`, `enabled=1`.

#### `routing_decisions`
Immutable log of every routing decision.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | TEXT | PK | UUID |
| `task_id` | TEXT | NOT NULL | `story_id` being routed |
| `task_type` | TEXT | NOT NULL | Stage/task type |
| `selected_agent` | TEXT | NOT NULL | Agent role chosen |
| `affinity_score` | REAL | NOT NULL | Affinity rank used (lower = better) |
| `effective_cost` | REAL | | `price × tokens / success_rate`; NULL for fallback |
| `cost_price` | REAL | | Price component |
| `cost_tokens` | INTEGER | | Token component |
| `cost_success_rate` | REAL | | Success rate component |
| `fallback_reason` | TEXT | | NULL, `affinity_fallback`, `no_data_fallback`, or `all_agents_failed` |
| `candidates_json` | TEXT | | JSON array of all scored candidates |
| `outcome` | TEXT | DEFAULT 'pending' | `pending`, `success`, `failed` |
| `decided_at` | TEXT | NOT NULL | ISO timestamp |

**Indexes:**
- `idx_routing_decisions_task_type ON routing_decisions(task_type)`
- `idx_routing_decisions_agent ON routing_decisions(selected_agent)`
- `idx_routing_decisions_decided_at ON routing_decisions(decided_at)`
- `idx_routing_decisions_task_id ON routing_decisions(task_id)`

#### `agent_performance_stats`
Materialized per-agent-per-task-type success statistics. Updated after each task completion.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | TEXT | PK | UUID |
| `agent_role` | TEXT | NOT NULL | Agent identifier |
| `task_type` | TEXT | NOT NULL | Task category |
| `total_attempts` | INTEGER | NOT NULL DEFAULT 0 | All-time attempts |
| `first_try_successes` | INTEGER | NOT NULL DEFAULT 0 | First-try successes (retry_count=0 at success) |
| `total_attempts_30d` | INTEGER | NOT NULL DEFAULT 0 | Rolling 30-day attempts |
| `first_try_successes_30d` | INTEGER | NOT NULL DEFAULT 0 | Rolling 30-day first-try successes |
| `avg_cost_usd` | REAL | | Average cost per run |
| `avg_tokens` | INTEGER | | Average tokens per run |
| `last_updated` | TEXT | NOT NULL | ISO timestamp |

**Indexes:**
- `UNIQUE(agent_role, task_type)`
- `idx_agent_perf_agent ON agent_performance_stats(agent_role)`

### No Altered Tables
Existing tables (`agent_runs`, `stories`) already have all required columns (`agent_role`, `cost_usd`, `input_tokens`, `output_tokens`, `status`, `retry_count`).

## Risk Assessment

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| 1 | **Cold start / no historical data** — all agents lack stats initially, causing universal fallback to `claude_code` | Medium | Seed `agent_performance_stats` from existing `agent_runs` data via a backfill query in the migration. Add a `bootstrap` flag that uses affinity rank alone until ≥10 data points exist per agent/task-type pair. |
| 2 | **Rolling window drift** — 30-day stats become stale if `refreshRollingWindows()` isn't called regularly | Low | Run refresh as a scheduled job (every 6 hours via `setInterval` in server startup, similar to existing `autoApproveSweep`). Stats are still usable between refreshes. |
| 3 | **Capability matrix misconfiguration** — operator disables all agents for a task type | Medium | Routing engine validates at least one agent is enabled per task type before saving. If all are disabled at query time, fallback to `claude_code` and log `all_agents_failed`. |
| 4 | **Backward compatibility** — existing scheduler behavior changes | Low | Routing engine is additive. If `capability_matrix` has no entry for a task type, fall through to the existing `getAgentRunner()` map. Feature flag `routing_engine_enabled` in `settings` table (default `1`) allows instant rollback. |
| 5 | **Performance — scoring query on every dispatch** | Low | Query hits 3 small tables (<100 rows each). SQLite prepared statements are sub-millisecond. No concern at current scale. |
| 6 | **Concurrent config updates** — two operators editing the same task type simultaneously | Low | SQLite WAL mode handles concurrent reads. Writes are serialized by SQLite. Last-write-wins is acceptable for config changes. UI shows `updated_at` timestamp to signal staleness. |

### What to Validate

- **Effective cost formula accuracy**: `price × tokens / success_rate` with edge cases:
  - `success_rate = 0` → treat as `Infinity` cost, agent ranks last
  - `success_rate = 1.0` → cost = `price × tokens` (best case)
  - Missing cost/token data → exclude from cost scoring, use affinity only
- **Hot-reload**: update matrix via API, immediately route a task, confirm new config is used (no cache staleness)
- **Decision log completeness**: every dispatch produces exactly one `routing_decisions` row with all fields populated
- **Backward compatibility**: disable routing engine via settings → scheduler uses legacy `getAgentRunner()` path unchanged
- **Migration idempotency**: migration 039 can run on a fresh DB and on an existing DB with data
- **Seed data accuracy**: backfill query produces correct stats from existing `agent_runs` history
