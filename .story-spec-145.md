# AOV-145: DDL migration: artifacts table with self-referential superseded_by FK

As an engineering agent, I need a database schema foundation for story artifacts so that subsequent API work can store typed artifact content with superseding support. This migration creates the artifacts table with all required columns including a self-referential superseded_by foreign key, enabling the system to track which artifact versions have been replaced. The table must support efficient lookups of non-superseded artifacts per story.


## Acceptance Criteria
- Given a fresh database, when the up migration is applied, then an artifacts table exists with columns: id (PK), story_id (FK to stories ON DELETE CASCADE), type (varchar with CHECK constraint allowing only: architecture, design, qa_report, pull_request, other), content (jsonb), superseded_by (nullable FK self-reference to artifacts.id), created_at (timestamptz defaulting to now())
- Given the artifacts table exists, when a database index is inspected, then a composite index on (story_id, superseded_by) exists to support efficient non-superseded artifact lookups
- Given a database with existing stories, when the migration is applied, then all existing story rows are preserved with full referential integrity intact
- Given the migration has been applied, when the down migration is run, then the artifacts table is dropped cleanly with no residual objects

# Architecture Specification — Artifacts Table Foundation

**Story:** DDL migration — `artifacts` table with self-referential `superseded_by` FK
**Date:** 2026-04-18

## Technical Approach

This story is a **backend-only DDL migration**. Scope: deliver a forward/reverse migration pair that creates the `artifacts` table, its CHECK constraint, FKs (including self-reference), and a composite index. No API, service, or UI code is written here — downstream stories will consume the schema. Migration runs via the project's existing migration runner (e.g. `node-pg-migrate` / Knex / custom `pg` runner). Idempotent, reversible, and safe on a DB containing existing `stories` rows.

Key decisions:
- `id` as `bigserial` PK (consistent with typical Node/PG patterns; adjust to `uuid` only if the repo standard dictates).
- `type` constrained via inline `CHECK` rather than a PG `ENUM` to avoid the operational pain of altering enums later.
- `superseded_by` nullable self-FK with `ON DELETE SET NULL` so deleting a newer version doesn't cascade-destroy older history.
- Composite index `(story_id, superseded_by)` supports the common predicate `WHERE story_id = $1 AND superseded_by IS NULL`.

## Data Model Changes

### New table: `artifacts`

| Column | Type | Constraints |
|---|---|---|
| `id` | `bigserial` | PRIMARY KEY |
| `story_id` | `bigint` | NOT NULL, FK → `stories(id)` ON DELETE CASCADE |
| `type` | `varchar(32)` | NOT NULL, CHECK (`type IN ('architecture','design','qa_report','pull_request','other')`) |
| `content` | `jsonb` | NOT NULL DEFAULT `'{}'::jsonb` |
| `superseded_by` | `bigint` | NULL, FK → `artifacts(id)` ON DELETE SET NULL |
| `created_at` | `timestamptz` | NOT NULL DEFAULT `now()` |

### Indexes

| Name | Definition | Purpose |
|---|---|---|
| `artifacts_pkey` | PK on `id` | implicit |
| `idx_artifacts_story_superseded` | `(story_id, superseded_by)` | non-superseded lookup per story |

## Risk Assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| Self-FK creates circular dependency on rollback | Low | `DROP TABLE` removes the self-FK with the table; tested in down migration |
| `ON DELETE CASCADE` on `story_id` destroys artifact history on story deletion | Medium | Documented; UX spec Q3 flags UX-level warning as follow-up |
| CHECK constraint on `type` blocks future types | Medium | Future additive migration: `ALTER TABLE ... DROP CONSTRAINT ... ADD CONSTRAINT ... CHECK (...)` — chose CHECK over ENUM specifically to keep this cheap |
| Composite index insufficient for other query shapes | Low | Partial index `WHERE superseded_by IS NULL` is an optimization; defer until profiling shows need |
| Large existing `stories` table slows migration | Low | Migration only creates a new empty table; no row-level work. Safe on any size |
| ID type mismatch with existing `stories.id` | High-impact if wrong | Verify `stories.id` type before writing migration; `story_id` must match exactly |
| Concurrent writers during migration | Low | `CREATE TABLE` takes ACCESS EXCLUSIVE briefly; negligible |

**Backward compatibility:** Purely additive. No existing code references `artifacts`. Rollback is clean.

### Integration tests (required)

Against a disposable Postgres (testcontainers / ephemeral DB):

1. **Up migration creates table correctly**
   - Query `information_schema.columns` — assert all 6 columns with correct types, nullability, defaults.
   - Query `information_schema.table_constraints` — assert PK, both FKs, CHECK constraint.
   - Query `pg_indexes` — assert `idx_artifacts_story_superseded` exists with correct column order.
2. **CHECK constraint enforced**
   - Insert row with `type='architecture'` → succeeds.
   - Insert row with `type='invalid'` → fails with check_violation.
3. **FK behavior**
   - Insert artifact referencing non-existent `story_id` → fails.
   - Delete parent story → child artifacts cascade-deleted.
   - Insert artifact A; insert artifact B with `superseded_by = A.id`; delete A → B.superseded_by becomes NULL.
4. **Self-reference**
   - Artifact can reference itself? Verify repo intent; recommend **not** allowed via app-layer check (DB allows it).
5. **Existing data preserved**
   - Seed `stories` with N rows, apply migration, assert `SELECT COUNT(*) FROM stories` unchanged.
6. **Down migration drops cleanly**
   - Apply up, apply down, assert `artifacts` absent from `pg_tables`, no orphan indexes/constraints.
7. **Idempotency of the pair**
   - up → down → up → down succeeds without error.
