# AOV-148: Dashboard indexes: CREATE INDEX CONCURRENTLY migration on runs/gates/stories

As a database administrator, I need four supporting indexes added to the runs, gates, and stories tables using CREATE INDEX CONCURRENTLY so that the dashboard aggregation query performs efficiently without locking writes or causing downtime during deployment.


## Acceptance Criteria
- Given the migration file migrations/20260418_dashboard_indexes.sql exists, when it is applied, then indexes are created on runs(project_id, status), runs(started_at DESC), gates(project_id, status) WHERE pending, and stories(project_id, created_at)
- When the migration runs, then all four indexes are created using CREATE INDEX CONCURRENTLY to avoid write locks on the affected tables
- When the migration is run a second time on a database where indexes already exist, then it completes without error (idempotent via IF NOT EXISTS or equivalent guard)
- Given the migration has been applied, when the dashboard aggregation query executes, then EXPLAIN ANALYZE shows index scans on the relevant tables instead of sequential scans

# Architecture Specification: Dashboard Indexes Migration

## Technical Approach

Single SQL migration file adding four supporting indexes to `runs`, `gates`, and `stories` tables using `CREATE INDEX CONCURRENTLY` to avoid blocking writes. All indexes guarded with `IF NOT EXISTS` for idempotency. No application code changes; performance improvement is transparent to ORM/query layer. Migration must run outside any transaction block (CONCURRENTLY requirement).

## Data Model Changes

No schema changes (no new tables, columns, or constraints). Four new indexes:

| Table | Index name | Columns | Type |
|---|---|---|---|
| `runs` | `idx_runs_project_status` | `(project_id, status)` | btree |
| `runs` | `idx_runs_started_at_desc` | `(started_at DESC)` | btree |
| `gates` | `idx_gates_project_status_pending` | `(project_id, status) WHERE status = 'pending'` | partial btree |
| `stories` | `idx_stories_project_created` | `(project_id, created_at)` | btree |

## API Changes

None. This is a pure infrastructure change.

## File Changes

**Create:**
- `migrations/20260418_dashboard_indexes.sql` — four `CREATE INDEX CONCURRENTLY IF NOT EXISTS` statements. Must not be wrapped in `BEGIN/COMMIT`. Add a header comment noting non-transactional execution.

**Modify:**
- Migration runner config (if the project uses `node-pg-migrate`, `knex`, or similar) — ensure this migration is flagged to run outside a transaction (e.g., `exports.config = { transaction: false }` for node-pg-migrate, or `knex.raw` with disableTransactions).

## Dependencies

None. Uses existing PostgreSQL + migration tooling.

## Risk Assessment

| # | Risk | Mitigation |
|---|---|---|
| 1 | `CREATE INDEX CONCURRENTLY` fails if run inside a transaction | Explicitly disable transaction wrapping in migration runner |
| 2 | Partial failure leaves an `INVALID` index | Re-running migration with `IF NOT EXISTS` skips valid ones; invalid indexes must be dropped manually (`DROP INDEX CONCURRENTLY`) before retry |
| 3 | Long build time on large tables blocks migration job timeout | Run during low-traffic window; monitor `pg_stat_progress_create_index` |
| 4 | Lock contention with long-running transactions delays index build | Identify and terminate blocking transactions before deploy |
| 5 | `status = 'pending'` literal may not match enum representation | Verify actual value (string vs enum) in `gates.status` column before finalizing partial predicate |

Backward compatibility: fully backward-compatible — indexes are additive. Rollback = `DROP INDEX CONCURRENTLY IF EXISTS` for each.

## Testing Strategy

**Pre-deploy validation:**
- Run migration against staging DB copy; verify all four indexes present via `\d+ runs`, `\d+ gates`, `\d+ stories`.
- Re-run migration; confirm no errors (idempotency).

**Query plan validation:**
- `EXPLAIN ANALYZE` the dashboard aggregation query before and after; confirm `Index Scan` / `Index Only Scan` replaces `Seq Scan` on each target table.
- Capture latency delta (expect <200 ms post-migration per UX spec).

**Integration:**
- CI migration test harness runs the SQL against ephemeral Postgres, then re-runs to assert idempotency.
- Smoke test dashboard endpoint post-deploy; assert p95 latency improvement.

**Unit:** N/A — no application code.

**Post-deploy monitoring:**
- Check `pg_stat_user_indexes.idx_scan` for each new index after 24h to confirm usage.
- Alert if any index shows `indisvalid = false` in `pg_index`.
