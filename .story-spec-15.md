# AOV-15: Extend ingestion with section-based PDF, per-sheet spreadsheet, and per-component design file parsing

As an agent consuming ingested artifacts, I need PDFs parsed into sections with per-section summaries, spreadsheets parsed per-sheet with schema summaries, and design files parsed per-component with descriptions — so that context retrieval can operate at a meaningful granularity rather than treating each file as a single opaque blob. Each extracted unit must have embeddings and summaries generated so it is fully retrievable via the context graph.


## Acceptance Criteria
- Given a PDF artifact, when ingested, then each logical section is extracted as a discrete node with a summary and embedding stored in pgvector
- Given a PDF with N sections, when ingested, then N section nodes are linked to the parent PDF artifact node in the context graph
- Given a spreadsheet artifact, when ingested, then each sheet is extracted as a discrete node with a schema summary (column names, types, row count) and embedding
- Given a spreadsheet with M sheets, when ingested, then M sheet nodes are linked to the parent spreadsheet artifact node
- Given a design file artifact, when ingested, then each component is extracted as a discrete node with a description and embedding
- Given a design file with K components, when ingested, then K component nodes are linked to the parent design artifact node
- All section/sheet/component nodes appear in embedding re-ranking results and are retrievable by the three-phase context retrieval pipeline
- Ingestion failures for individual sections/sheets/components are isolated — a single parse error does not abort the entire file ingestion
- Ingestion pipeline reports per-file counts of extracted units (sections, sheets, components) in its response or logs

## ARCHITECTURE

Architecture spec written to `/app/.arch-spec-36421a6b-architecture.md`.

**Summary of key decisions:**

- **SQLite → PostgreSQL migration required** — pgvector has no SQLite equivalent; this is the highest-risk item and may warrant a separate prerequisite story
- **3 new tables**: `parsed_units` (with `vector(1536)` column), `context_graph_nodes`, `context_graph_edges`

- **5 new API endpoints**: unit listing, unit detail, graph nodes, graph edges, context retrieval (three-phase)
- **17 new files, 7 modified files** across backend services, parsers, API routes, and React components
- **5 risks identified** — PG migration (high), PDF section detection (medium), design file formats (medium), embedding cost/latency (medium), graph density (low)
- **7 new dependencies**: `pg`, `pgvector`, `pdf-parse`, `xlsx`, embedding SDK, `react-virtuoso`, `react-force-graph-2d`
