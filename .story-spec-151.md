# AOV-151: DDL migration: complexity/file_count columns, architecture artifact type, architect agent type, and indexes

As a developer working on the architect agent pipeline, I need the database schema extended with complexity and file_count columns on the stories table, an 'architecture' value in the artifacts.type enum, an 'architect' value in the agent_runs.agent_type enum, and supporting indexes so that architect agent runs can persist structured metadata and artifacts correctly.


## Acceptance Criteria
- Given the migration runs up, the stories table has a nullable complexity column with CHECK constraint limiting values to ('low','medium','high','epic') and a nullable integer file_count column

- Given the migration runs up, the agent_runs.agent_type enum includes 'architect' as a valid value alongside existing types

- Given the migration runs down, all added columns, enum values, and indexes are cleanly removed and the schema is restored to its prior state

## ARCHITECTURE

# Architecture Specification: DDL Migration for Architect Agent Pipeline

## Technical Approach

Pure backend schema migration. No API, no UI, no runtime code changes. Deliver a single forward/rollback migration pair that:

1. Adds `complexity` (text, CHECK-constrained) and `file_count` (int) nullable columns to `stories`.
2. Extends the `artifacts.type` enum with `'architecture'`.
3. Extends the `agent_runs.agent_type` enum with `'architect'`.
4. Creates `idx_artifacts_story_type` and `idx_agent_runs_story_status`.

Use raw SQL migrations (not an ORM DSL) because PostgreSQL enum additions require `ALTER TYPE ... ADD VALUE` which cannot run inside a transaction when paired with other DDL that depends on the new value. Split into clearly-ordered statements; keep enum additions idempotent via `IF NOT EXISTS` where supported.

For rollback: enum values cannot be dropped directly in Postgres â€” the down migration must recreate the enum type without the added value, update dependent columns to use the new type, and drop the old type. Document this explicitly in migration comments.

## Data Model Changes

### `stories` table â€” new columns

| Column | Type | Null | Constraint |
|---|---|---|---|
| `complexity` | `TEXT` | YES | `CHECK (complexity IN ('low','medium','high','epic'))` |
| `file_count` | `INTEGER` | YES | `CHECK (file_count >= 0)` (recommended) |

### `artifacts.type` enum

- Add value: `'architecture'`
- Existing values preserved (ordering appended at end).

### `agent_runs.agent_type` enum

- Add value: `'architect'`
- Existing values preserved.

### New indexes

| Index | Table | Columns | Purpose |
|---|---|---|---|
| `idx_artifacts_story_type` | `artifacts` | `(story_id, type)` | Lookup artifacts by story filtered by type (e.g., fetch architecture artifact for a story) |
| `idx_agent_runs_story_status` | `agent_runs` | `(story_id, status)` | Find active/completed runs for a story â€” hot path for scheduler |

### Migration SQL (sketch)

**Up:**
```sql
ALTER TABLE stories
  ADD COLUMN complexity TEXT
    CHECK (complexity IN ('low','medium','high','epic')),
  ADD COLUMN file_count INTEGER CHECK (file_count >= 0);

ALTER TYPE artifact_type ADD VALUE IF NOT EXISTS 'architecture';
ALTER TYPE agent_type    ADD VALUE IF NOT EXISTS 'architect';

CREATE INDEX IF NOT EXISTS idx_artifacts_story_type
  ON artifacts (story_id, type);
CREATE INDEX IF NOT EXISTS idx_agent_runs_story_status
  ON agent_runs (story_id, status);
```

**Down:** (enum removal requires type rebuild)
```sql
DROP INDEX IF EXISTS idx_agent_runs_story_status;
DROP INDEX IF EXISTS idx_artifacts_story_type;

ALTER TABLE stories
  DROP COLUMN file_count,
  DROP COLUMN complexity;

-- Rebuild artifact_type without 'architecture'
ALTER TYPE artifact_type RENAME TO artifact_type__old;
CREATE TYPE artifact_type AS ENUM (/* original values */);
ALTER TABLE artifacts
  ALTER COLUMN type TYPE artifact_type
  USING type::text::artifact_type;
DROP TYPE artifact_type__old;

-- Same pattern for agent_type
ALTER TYPE agent_type RENAME TO agent_type__old;
CREATE TYPE agent_type AS ENUM (/* original values */);
ALTER TABLE agent_runs
  ALTER COLUMN agent_type TYPE agent_type
  USING agent_type::text::agent_type;
DROP TYPE agent_type__old;
```

## Risk Assessment

| # | Risk | Mitigation |
|---|---|---|
| 1 | **Enum rollback complexity** â€” Postgres does not support `DROP VALUE` on enums. Down migration must rebuild type. | Script the rename/recreate/swap pattern explicitly; test down migration in CI against a populated DB. |
| 2 | **Index creation locks** â€” `CREATE INDEX` on large `artifacts`/`agent_runs` tables may block writes. | If tables are large in prod, switch to `CREATE INDEX CONCURRENTLY` (must run outside transaction). For dev-stage repo, plain `CREATE INDEX` is fine. |
| 3 | **CHECK constraint on existing rows** â€” Not a risk here (column is new and nullable) but flagged for future backfills. | Any future backfill must validate against the CHECK set before writing. |
| 4 | **Concurrent migration runs** â€” Two schedulers running migrations simultaneously could race. | Rely on migration tool's advisory lock; verify it is enabled. |
| 5 | **Dependent code assumes old enum values** â€” Any switch/exhaustive match on `artifact_type` or `agent_type` in TS code will now be non-exhaustive. | Grep for switch statements on these enums; add default branches or update union types in a follow-on PR. Not blocking for this migration. |

### Backward compatibility
