# AOV-16: Auto edge creation for context graph on ingestion

As a developer using the context graph system, I need edges between artifacts to be automatically created during ingestion so that the graph reflects real relationships without manual linking. This includes import-based dependency edges (imports→depends_on), wiki-style link extraction from markdown (wiki-links→references), embedding similarity edges for semantically related content (cosine similarity > 0.7 → related_to), and directory hierarchy edges (child_of). All edge types should be derived and written to the graph at ingestion time, enabling downstream features like context retrieval and graph exploration to leverage a rich, automatically-populated relationship graph.


## Acceptance Criteria
- Given a code file is ingested, When it contains import/require statements, Then a depends_on edge is created from the artifact to each imported artifact that exists in the graph
- Given a markdown file is ingested, When it contains wiki-style links (e.g. [[PageName]]), Then a references edge is created from the artifact to the linked artifact if it exists in the graph
- Given any artifact is ingested with an embedding, When its cosine similarity to an existing artifact's embedding exceeds 0.7, Then a related_to edge is created between the two artifacts
- Given any artifact is ingested, When it resides in a directory that maps to a parent artifact in the graph, Then a child_of edge is created from the artifact to its parent directory artifact
- Given explicit links are present in the JSON interchange format, When an artifact is ingested, Then those explicit edges are created in addition to auto-derived edges
- Given ingestion completes, When edges are written, Then all edge types (depends_on, references, related_to, child_of) are stored with the correct type label and source/target artifact IDs in the graph schema

# Architecture Spec: Auto Edge Creation for Context Graph on Ingestion

## Technical Approach

Extend the existing artifact ingestion pipeline with an **edge derivation engine** that automatically creates typed graph edges when artifacts are ingested. The system hooks into the artifact creation flow (POST `/api/stories` artifact writes and agent artifact outputs) and runs four derivation strategies in sequence:

1. **Import Parser** — Static analysis of `import`/`require` statements in code artifacts → `depends_on` edges
2. **Wiki Link Extractor** — Regex extraction of `[[PageName]]` patterns from markdown artifacts → `references` edges
3. **Embedding Similarity** — Cosine similarity comparison against existing artifact embeddings (threshold > 0.7) → `related_to` edges
4. **Directory Hierarchy** — Path-based parent resolution from artifact URL/path → `child_of` edges

Additionally, **explicit edges** provided in the JSON interchange payload are written directly.

All derivation runs within an **ingestion job** that tracks per-edge-type status and counts, exposed via WebSocket for real-time UI updates through the existing `broadcaster.js` infrastructure.

**Open question resolutions:**
1. Similarity scores displayed to 2 decimal places (0.73) — more useful for developers
2. Explicit edges that duplicate auto-derived edges are merged; `derived_from` = `"explicit"` takes precedence
3. Real-time push via existing WebSocket (`/ws`) — no polling needed
4. Similarity threshold persisted in `localStorage` per user session
5. Orphan edges (dangling target) rendered with strikethrough label + warning icon; not auto-deleted

| `GET` | `/api/graph/edges` | Paginated edge list for review table | `?edge_type=X&source=X&target=X&min_similarity=X&max_similarity=X&page=1&per_page=50&sort_by=created_at&sort_dir=desc` | `{ edges: [...], total, page, per_page }` |

| `GET` | `/api/graph/artifacts/:id/edges` | Edges for a single artifact (drawer) | `?direction=outgoing|incoming|both` | `{ outgoing: [...], incoming: [...] }` |

## Risk Assessment

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **Embedding similarity is O(n) per ingestion** — comparing a new artifact against all existing embeddings grows linearly | Medium | Batch comparisons in chunks of 100. Add a `project_id` filter to scope comparisons. For >10k artifacts, consider approximate nearest neighbors (future migration to `sqlite-vss`). |
| 2 | **Circular edge derivation** — ingesting artifact A creates edges, which could re-trigger derivation | Low | `edgeDerivation.js` only runs on explicit ingest triggers (POST), not on edge creation. Guard with `ingestion_jobs` dedup check per artifact. |
| 3 | **Import path resolution ambiguity** — `import foo from './bar'` may resolve to `bar.js`, `bar/index.js`, `bar.ts`, etc. | Medium | `importParser.js` attempts resolution with common extensions (`['.js','.ts','.jsx','.tsx','']`). Falls back to fuzzy match on artifact path suffix. Unresolved imports are logged but don't create edges. |
| 4 | **Large graph rendering performance** — force-directed layout with >500 nodes can lag | Medium | `react-force-graph-2d` uses canvas (not SVG). Add node limit with pagination / viewport culling. Default to showing only 1-hop neighborhood of selected node. |
| 5 | **Unique constraint on duplicate edges** — explicit edge matching an auto-derived edge hits the unique index | Low | Use `INSERT OR REPLACE` (SQLite upsert). Explicit edges overwrite auto-derived edges for the same `(source, target, type)` tuple, updating `derived_from` to `'explicit'`. |
