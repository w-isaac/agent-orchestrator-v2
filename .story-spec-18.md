# AOV-18: Token Analytics Dashboard: Charts, Gauges, and Cost Visualizations

As a project operator or team lead, I want a comprehensive token analytics dashboard so that I can monitor token consumption trends, understand agent efficiency, identify rework costs, and track budget utilization in real time. The dashboard should surface actionable metrics—usage by project, agent, and time period; first-try success rates; rework cost breakdowns; and conflict rates—so I can make informed decisions about agent routing, budget allocation, and quality improvement.


## Acceptance Criteria
- Given the dashboard is open, when I select a time range, then token usage charts update to show consumption broken down by project, agent, and time bucket (hourly/daily/weekly)
- Given the dashboard is open, then a first-try success rate metric is displayed per agent and per project, calculated as tasks completed without rework divided by total tasks
- Given tasks with rework exist, when I view the rework cost panel, then I see the total tokens and estimated dollar cost attributed to rework (re-dispatched tasks) grouped by agent and project
- Given the dashboard is open, then an effective cost breakdown panel shows total spend decomposed by agent model, task type, and project
- Given a budget has been configured per project, when I view the budget utilization gauges, then each gauge shows current spend vs. budget cap with visual warning thresholds at 75% and 90%
- Given conflict resolution events exist, when I view the conflict rate metrics panel, then I see conflict counts, auto-merge rates, and manual-intervention rates over the selected time range
- Given I am on the dashboard, when I hover over any chart data point, then a tooltip shows the exact value, timestamp, and contributing entity (agent or project)
- Given the analytics API returns no data for a selected filter, then the dashboard displays an empty-state message rather than broken charts

# Architecture Spec: Token Analytics Dashboard

## Technical Approach

Build a dedicated analytics subsystem on top of the existing `agent_runs`, `stories`, and `system_events` tables. The existing data model already captures token counts, USD cost, model attribution, retry counts, and QA bounces — most analytics queries are pure aggregations over `agent_runs` joined to `stories`.

**Key decisions:**
- **No new data warehouse** — SQLite with WAL mode handles analytical reads concurrently with writes at this scale
- **One new table** — `project_budgets` for budget cap configuration (no budget storage exists today)
- **New route group** — `POST /api/analytics/*` endpoints serve pre-aggregated JSON to the frontend; keeps query logic server-side and cacheable
- **Conflict tracking** — leverage `system_events` (type/subtype) for conflict resolution events; add a new system event subtype if the deployment agent doesn't already emit them
- **Frontend** — new React components under `client/src/components/analytics/`; depends on Recharts for charting
- **Polling, not WebSocket** — dashboard refreshes on filter apply (user-driven); no auto-poll in v1 (see Open Question #3 resolution below)

## Data Model Changes

### New Table: `project_budgets`

```sql
-- Migration: 039_project_budgets.sql
CREATE TABLE IF NOT EXISTS project_budgets (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL UNIQUE REFERENCES projects(id) ON DELETE CASCADE,
  budget_cap_usd   REAL NOT NULL,
  period           TEXT NOT NULL DEFAULT 'monthly',  -- 'weekly' | 'monthly'
  period_start     TEXT,  -- ISO date; NULL = auto (first of month / Monday)
  created_at       TEXT NOT NULL,
  updated_at       TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_project_budgets_project ON project_budgets(project_id);
```

### Existing Tables Used (no schema changes)

| Table | Columns Used | Purpose |
|-------|-------------|---------|
| `agent_runs` | `cost_usd`, `input_tokens`, `output_tokens`, `cache_read_tokens`, `cache_write_tokens`, `claude_model`, `agent_role`, `project_id`, `story_id`, `started_at`, `finished_at`, `status` | Token usage, cost breakdowns, model attribution |
| `stories` | `retry_count`, `qa_bounce_count`, `project_id`, `status` | First-try rate, rework identification |
| `system_events` | `type`, `subtype`, `payload`, `created_at` | Conflict resolution events (subtype: `conflict_detected`, `auto_merge`, `manual_intervention`) |
| `projects` | `id`, `name` | Labels for dropdowns and charts |
| `agent_roster` | `role`, `name` | Agent display names |
