# AOV-183: Analytics DB indexes, materialized view, and hourly refresh job

As a developer building analytics endpoints, I need a reliable analytics foundation in the database so that query performance is acceptable and daily run metrics are available without recomputing on every request. This story lays the additive groundwork: performance indexes on the runs and stories tables, a materialized view mv_daily_run_metrics pre-aggregating daily statistics, and a scheduled hourly refresh job wired into the server so the view stays reasonably fresh. No existing tables are altered; all changes are purely additive and reversible.


## Acceptance Criteria
- Given migration 20260418_analytics_indexes.sql is applied, then all 5 documented indexes exist on the runs and stories tables and queries against those columns use index scans
- Given migration 20260418_create_mv_daily_run_metrics.sql is applied, then the mv_daily_run_metrics materialized view exists with all documented columns and can be queried without error even when empty
- Given both migrations have been applied, when REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_run_metrics is executed, then it completes successfully and the view reflects current data
- Given the server starts, then refreshAnalyticsMv.js is registered as an hourly cron job in server/src/app.js and runs without error when the MV is empty
- Given both migrations are rolled back (or are purely additive drop-safe), then no existing table columns or data are lost

# Architecture Specification: Analytics DB Indexes, Materialized View, and Hourly Refresh Job

## Technical Approach

Purely additive backend/infrastructure change across three layers:

1. **DB migrations (additive only)** â€” Two forward-only SQL migrations: one adding 5 btree indexes to `runs` and `stories`, and one creating the `mv_daily_run_metrics` materialized view with a unique index enabling `REFRESH ... CONCURRENTLY`.
2. **Server cron module** â€” A new `server/src/jobs/refreshAnalyticsMv.js` exposing a `register(app)` function that schedules an hourly `REFRESH MATERIALIZED VIEW CONCURRENTLY` using `node-cron`.
3. **App wiring** â€” `server/src/app.js` imports and invokes the registration function during startup (after DB pool is ready, before server listen).

Non-blocking semantics: refresh runs in background; failures are logged, not fatal. First refresh is triggered on startup (unless MV empty and unique index missing â€” handled).

| `idx_stories_status` | btree(stories.status) | Status filtering |

### Materialized view shape

```sql
CREATE MATERIALIZED VIEW mv_daily_run_metrics AS
SELECT
  date_trunc('day', r.created_at)::date AS day,
  COUNT(*)                              AS total_runs,
  COUNT(*) FILTER (WHERE r.status='success') AS success_runs,
  COUNT(*) FILTER (WHERE r.status='failed')  AS failed_runs,
  AVG(EXTRACT(EPOCH FROM (r.completed_at - r.created_at))) AS avg_duration_seconds,
  COUNT(DISTINCT r.story_id)            AS distinct_stories
FROM runs r
GROUP BY 1
WITH NO DATA;

CREATE UNIQUE INDEX mv_daily_run_metrics_day_uidx
  ON mv_daily_run_metrics(day);
```

## Risk Assessment

| # | Risk | Mitigation |
|---|---|---|
| 1 | `REFRESH CONCURRENTLY` fails when MV has never been populated | Issue a one-time non-concurrent `REFRESH MATERIALIZED VIEW` on first startup, or catch error and fall back |
| 2 | `CREATE INDEX CONCURRENTLY` cannot run inside a transaction â€” may conflict with migration runner that wraps in BEGIN/COMMIT | Use plain `CREATE INDEX IF NOT EXISTS` unless runner supports concurrent mode; document tradeoff (brief lock on large tables) |
| 3 | Overlapping refresh runs if one hour exceeds refresh time | In-memory boolean lock in `refreshAnalyticsMv.js`; skip new tick if prior still running |
| 4 | Cron registered in test env pollutes integration tests | Gate registration behind env check |
| 5 | MV definition assumes columns `runs.created_at`, `runs.completed_at`, `runs.status`, `runs.story_id` â€” verify schema before finalizing migration | Inspect `runs` table schema; adjust column names to actuals before merge |
| 6 | Index bloat / write amplification on `runs` / `stories` | Acceptable for analytics workload; monitor via `pg_stat_user_indexes` |

**Backward compatibility:** Fully compatible. No column adds/drops, no renames, no altered types. Rolling back is safe (`DROP` statements only).

## Testing Strategy

### Migration tests
- Apply `20260418_analytics_indexes.sql` against a fresh test DB; assert all 5 indexes exist via `pg_indexes`.
- Apply MV migration; assert MV exists via `pg_matviews` and columns match expected shape.
- Run `REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_run_metrics` against empty `runs` â€” must succeed (the unique index + empty refresh is the known edge case).
- Seed a few rows into `runs`; refresh; assert row counts/aggregates in MV.

### Unit tests (`server/src/jobs/refreshAnalyticsMv.test.js`)
- `register()` schedules a cron with expression `0 * * * *`.
- Refresh function calls `pool.query` with the expected SQL.
- Overlap lock: two concurrent invocations result in only one DB call.
- Error path: DB error is logged, does not throw out of the scheduled tick.

### Integration tests
- Start the app with a test DB; assert cron is registered (spy/mock `node-cron.schedule`).
- Manually invoke the refresh handler; verify MV contents update.
- Assert app does not register cron when `NODE_ENV==='test'` (or flag disabled).

### Query plan validation
- Run `EXPLAIN` on representative analytics queries filtering by `runs.created_at`, `runs.status`, `runs.story_id`, `stories.created_at`, `stories.status`; confirm **Index Scan** (not Seq Scan) is used on a populated test dataset.

### Rollback test
- Apply then drop MV and indexes; confirm `runs` and `stories` row counts and column lists are unchanged.
