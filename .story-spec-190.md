# AOV-190: Iterations table migration and model

As a backend engineer, I need the iterations table created in the database with all required columns, constraints, indexes, and an updated_at trigger, plus a server-side model module exposing typed query functions. This is the foundational data layer that the REST endpoints (story #191), SSE stream (story #192), and frontend (story #193) all depend on. Without this migration and model in place, no other iterations work can begin.


## Acceptance Criteria
- Migration creates the iterations table with all specified columns (id, story_id, iteration_number, status, created_at, updated_at, and any payload columns) and a CHECK constraint on the status column
- Indexes idx_iterations_story_id_created_at (composite) and uniq_iterations_story_number (unique) are created by the migration
- A database trigger automatically updates the updated_at column on every row UPDATE
- The model module exports list(storyId, { limit, offset, order }), create(data) (auto-increments iteration_number transactionally), getById(id), and getLatestWithQAFailure(storyId) query functions with correct TypeScript types
- Unit tests cover concurrent create() calls to verify uniqueness constraint enforcement and verify list() returns results in the correct order

# Architecture Specification — Iterations Table Migration & Model (Story #190)

## Technical Approach

Foundational data-layer story. Deliver (1) a forward-only SQL migration creating the `iterations` table with constraints/indexes/trigger, and (2) a typed Node.js model module exposing four query functions. No HTTP endpoints, no React components — those are downstream stories (#191, #192, #193). The model must guarantee monotonic `iteration_number` per `story_id` under concurrent writes via a serializable-safe transaction pattern backed by a DB-level unique constraint.

**Key design decisions:**
- `iteration_number` computed inside a transaction using `SELECT COALESCE(MAX(iteration_number),0)+1 ... FOR UPDATE` on sibling rows, with the unique index as the ultimate race guard.
- `payload` stored as `JSONB` (unenumerated per open question #1) — flexible, indexable, forward-compatible.
- Status enum enforced via `CHECK` constraint (not a PG `ENUM` type) to keep migrations additive-friendly.
- `updated_at` maintained via `BEFORE UPDATE` trigger using a shared `set_updated_at()` function (create if not present).

---

## Data Model Changes

### New table: `iterations`

| Column | Type | Constraints |
|---|---|---|
| `id` | `BIGSERIAL` | PK |
| `story_id` | `BIGINT` | NOT NULL, FK → `stories(id)` ON DELETE CASCADE |
| `iteration_number` | `INTEGER` | NOT NULL, CHECK (> 0) |
| `status` | `TEXT` | NOT NULL, CHECK IN (`pending`, `in_progress`, `completed`, `qa_failed`, `cancelled`) |
| `payload` | `JSONB` | NOT NULL DEFAULT `'{}'::jsonb` |
| `created_at` | `TIMESTAMPTZ` | NOT NULL DEFAULT `now()` |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL DEFAULT `now()` |

### Indexes
- `idx_iterations_story_id_created_at` — composite `(story_id, created_at DESC)` — supports `list()` ordering.
- `uniq_iterations_story_number` — UNIQUE `(story_id, iteration_number)` — race guard.

### Trigger
- `trg_iterations_updated_at` — `BEFORE UPDATE FOR EACH ROW` → `set_updated_at()` (sets `NEW.updated_at = now()`).

### Migration
- Single forward migration file (timestamped). Creates shared `set_updated_at()` function `IF NOT EXISTS` via `CREATE OR REPLACE FUNCTION`. Down-migration drops table, trigger, and (only its own) index artifacts.

---

## API Changes

**No HTTP endpoints in this story.** The model module contract (consumed by #191, #192, #193):

| Function | Signature | Returns |
|---|---|---|
| `list` | `list(storyId: number, { limit=20, offset=0, order='desc' })` | `Promise<Iteration[]>` |
| `create` | `create({ storyId: number, status?: IterationStatus, payload?: object })` | `Promise<Iteration>` (auto-assigns `iteration_number`, default status `pending`) |
| `getById` | `getById(id: number)` | `Promise<Iteration \| null>` |
| `getLatestWithQAFailure` | `getLatestWithQAFailure(storyId: number)` | `Promise<Iteration \| null>` (most recent row where `status='qa_failed'`) |

**Error contract:** `create()` surfaces unique-violation (`23505`) as a typed `IterationConflictError` so story #191 can map to HTTP 409.

## Risk Assessment

| # | Risk | Mitigation |
|---|---|---|
| 1 | Concurrent `create()` races → duplicate `iteration_number` | `FOR UPDATE` lock on sibling rows inside txn + unique index as last-line guard; retry once on `23505` |
| 2 | Status enum drift between CHECK constraint and TS union | Single source: generate TS union from a constant array also referenced in a CHECK-constraint assertion test |
| 3 | `payload` schema unspecified (open Q1) | `JSONB` default — accepts any shape; defer validation to caller/story #191 |
| 4 | Trigger recursion if `updated_at` set in `UPDATE … SET updated_at=…` | Trigger unconditionally overwrites; no recursion possible (BEFORE UPDATE, single assignment) |
| 5 | `FOR UPDATE` on sibling rows blocks under high write volume | Acceptable for per-story scope; document; revisit if throughput becomes an issue |
| 6 | Backward compatibility — table is new | None — additive migration only |

**Open questions deferred to product:** payload columns (Q1), status enum final list (Q2), retry semantics (Q4). Current spec assumes answers given in the UX doc until clarified.

## Testing Strategy

### Unit (type-level)
- TS type assertions on model signatures (`expectType<Iteration>`).
- Status union ↔ CHECK-constraint array parity test.

### Integration (real Postgres, transactional fixtures)
- **Migration:** table, indexes, trigger, CHECK constraint exist post-migrate; down-migration is clean.
- **`create()` happy path:** inserts row, `iteration_number=1` for first, `2` for second.
- **Concurrency:** spawn N=20 parallel `create()` calls for same `storyId` via `Promise.all` — all succeed, `iteration_number` set is `{1..20}` with no gaps or duplicates.
- **Unique constraint enforcement:** force-insert duplicate `(story_id, iteration_number)` bypassing model → expect `23505`.
- **`list()` ordering:** insert 5 rows with staggered timestamps; assert `desc` default and `asc` override.
- **`list()` pagination:** `limit`/`offset` behavior.
- **Trigger:** `UPDATE` bumps `updated_at`; `created_at` unchanged.
- **`getById`:** hit + miss (null).
- **`getLatestWithQAFailure`:** returns most recent `qa_failed`; returns null when none; ignores other statuses.
- **Error mapping:** `IterationConflictError` thrown on simulated `23505`.

### Out of scope
- HTTP/SSE/UI tests — belong to #191, #192, #193.
