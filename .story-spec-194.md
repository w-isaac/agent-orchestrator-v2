# AOV-194: Smoke config table, CRUD endpoints, and synchronous connectivity test

As a platform engineer, I need a smoke_configs table and backend API so that each project can store a base_url and a set of named routes to probe, and I can verify connectivity synchronously before committing to a full smoke run. This story covers only the data layer and HTTP endpoints — no frontend, no run execution. The migration creates the table with a unique constraint on project_id and a jsonb routes column. Three endpoints are required: GET to retrieve the current config (or 404), POST to upsert base_url and routes with Zod validation, and a synchronous POST /test that probes base_url and returns reachability, HTTP status, latency, and any error string.


## Acceptance Criteria
- Given the database is migrated, a smoke_configs table exists with columns id, project_id (unique FK), base_url, routes (jsonb), created_at, updated_at
- When GET /api/projects/:id/smoke-config is called for a project with no config, Then the response is 404
- When GET /api/projects/:id/smoke-config is called for a project with an existing config, Then it returns the stored base_url and routes with 200
- When POST /api/projects/:id/smoke-config is called with a valid base_url and routes array, Then it upserts the record and returns the saved config with 200
- When POST /api/projects/:id/smoke-config is called with an invalid method, empty path, or negative timeout in routes, Then Zod validation rejects the request with 422 and a descriptive error
- When POST /api/projects/:id/smoke-config/test is called, Then the endpoint probes base_url synchronously and returns { reachable: boolean, status: number | null, latency_ms: number, error: string | null } with 200


## Project Infrastructure Settings

- **deploy_port**: 5001 (use this as the host port in docker-compose.yml: `5001:3000`)
- **container_port**: 3000

## ARCHITECTURE

# Architecture Specification — Smoke Config API

## Technical Approach

Backend-only delivery. Add a single `smoke_configs` table with a unique FK to `projects`, three Express routes under `/api/projects/:id/smoke-config`, and Zod-validated upsert semantics. The `/test` endpoint performs a synchronous HTTP probe (GET) against `base_url` using the global `fetch` with an AbortController timeout and returns a normalized result envelope. No queues, no background jobs, no frontend.

**Stack alignment:** Node ≥20 (native `fetch`), Express router module, pg via existing DB pool, Zod schemas mirrored to TypeScript types, Knex/node-pg-migrate (match existing project convention) for migration.

---

## Data Model Changes

### New table: `smoke_configs`

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | `gen_random_uuid()` default |
| `project_id` | `uuid` NOT NULL | FK → `projects(id)` ON DELETE CASCADE, **UNIQUE** |
| `base_url` | `text` NOT NULL | validated `https?://` at app layer |
| `routes` | `jsonb` NOT NULL | default `'[]'::jsonb` |
| `created_at` | `timestamptz` NOT NULL | default `now()` |
| `updated_at` | `timestamptz` NOT NULL | default `now()`, trigger-updated |

### Indexes
- `UNIQUE (project_id)` — enforces one config per project and powers upsert.
- FK index implicit via unique constraint.

### Migration
`migrations/NNNN_create_smoke_configs.sql` — creates table, unique constraint, `updated_at` trigger (reuse existing `set_updated_at` function if present; else inline).

### Route JSONB shape (app-validated, not DB-enforced)
```json
[{ "name": "health", "method": "GET", "path": "/healthz", "timeout_ms": 5000 }]
```

---

## API Changes

All routes mounted at `/api/projects/:id/smoke-config`.

### 1. GET `/api/projects/:id/smoke-config`
- **Response 200:** `{ id, project_id, base_url, routes, created_at, updated_at }`
- **Response 404:** `{ error: "smoke_config_not_found" }`

### 2. POST `/api/projects/:id/smoke-config`
- **Request body:**
  ```ts
  {
    base_url: string (url, http/https),
    routes: Array<{
      name: string (1..64),
      method: "GET"|"POST"|"PUT"|"DELETE"|"PATCH",
      path: string (startsWith "/"),
      timeout_ms?: number (int, 1..60000)
    }>
  }
  ```
- **Response 200:** saved config row (upsert via `INSERT ... ON CONFLICT (project_id) DO UPDATE`).
- **Response 422:** `{ error: "validation_failed", issues: ZodIssue[] }`
- **Response 404:** project does not exist.

### 3. POST `/api/projects/:id/smoke-config/test`
- **Request body:** none (uses stored `base_url`; 404 if no config).
- **Response 200:**
  ```ts
  { reachable: boolean, status: number | null, latency_ms: number, error: string | null }
  ```
- **Probe semantics:** `GET base_url` with `AbortSignal.timeout(10000)`; `reachable = status !== null && status < 500`; record wall-clock latency; map fetch errors to stable strings (`"connection_refused"`, `"dns_error"`, `"timeout"`, `"tls_error"`, fallback `err.message`).

## Risk Assessment

| # | Risk | Mitigation |
|---|---|---|
| 1 | **SSRF** — `base_url` could point at `169.254.169.254`, `localhost`, private ranges | Validate against a denylist of private/link-local CIDRs before probing; document residual risk |
| 2 | **Probe hangs** tie up event loop / socket | Enforce `AbortSignal.timeout(10000)` hard cap; do not follow excessive redirects (`redirect: "manual"` or cap) |
| 3 | **Route JSONB schema drift** — DB accepts anything | Zod is the single source of truth on write; add defensive parse on read to surface legacy rows |
| 4 | **Concurrent upserts** on same project | `ON CONFLICT (project_id) DO UPDATE` is atomic; last-write-wins is acceptable per AC |
| 5 | **Unbounded routes array / giant payloads** | Cap `routes.length ≤ 50`; cap body size via existing `express.json({ limit: '64kb' })` |
| 6 | **Error string leakage** in `/test` response | Map to normalized strings, never expose stack traces or internal hostnames |

**Backward compatibility:** New table, new routes — no breaking changes.

## Testing Strategy

### Unit
- `smokeConfigSchema`: accepts valid payload; rejects invalid method, empty path, negative `timeout_ms`, non-URL `base_url`, oversize routes array.
- `probe.ts`: mock `fetch` to simulate 200, 500, network error, timeout → assert envelope shape and latency is a non-negative integer.

### Integration (supertest + test DB)
- GET returns 404 when no row exists.
- POST inserts then GET returns same payload.
- POST a second time upserts (same `id`, updated `updated_at`, new `routes`).
- POST with invalid body → 422 with Zod `issues`.
- POST `/test` against a local fixture server: reachable true/200/latency > 0; unreachable (closed port) → `reachable:false, status:null, error:"connection_refused"`; slow server → `error:"timeout"`.
- Unique constraint enforced: direct DB insert of duplicate `project_id` fails.

### E2E / Acceptance
- Walk each of the 6 acceptance criteria as a scripted test and assert status code + body shape.

### What to validate
- Migration is idempotent and reversible (`down` drops table + trigger).
- `updated_at` changes on upsert.
- 422 error body is stable (contract for future frontend).
- `/test` never throws — always returns the envelope with 200.

## User Flow

1. User navigates to **Project → Settings → Smoke Config**.
2. If no config exists: sees empty state → clicks **Add Config**.
3. Enters `base_url` (e.g., `https://api.myapp.com`).
4. Clicks **Add Route** → fills in name, method, path, optional timeout.
5. Repeats step 4 for additional routes.
6. Clicks **Test Connection** → inline result panel expands below button.
7. Reviews reachability, status, latency, and error.
8. If satisfied, clicks **Save Config** → success toast "Smoke config saved".
9. On return visits: form pre-filled from GET response; user edits and re-saves.
