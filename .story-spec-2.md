# AOV-2: Set up core Postgres schema with pgvector and all indexes

As a backend engineer, I need the foundational database schema created so that all other services have a stable, indexed data layer to build upon. This includes all tables defined in the architecture document (Section 5.1): context_nodes, context_edges, tasks, task_results, node_locks, task_snapshots, and projects. The pgvector extension must be enabled to support embedding-based similarity search. All required indexes — covering project, type, embedding (ivfflat), and edge traversal (source/target/type) — must be in place. Seed data for local development and testing should also be included.


## Acceptance Criteria
- Given the migrations run successfully, all seven tables exist: context_nodes, context_edges, tasks, task_results, node_locks, task_snapshots, projects
- Given the schema is applied, the pgvector extension is enabled and the embedding column on context_nodes uses a vector type
- Given the schema is applied, an ivfflat index exists on the embedding column to support approximate nearest-neighbour queries
- Given the schema is applied, indexes exist on context_nodes for project_id and type columns, and on context_edges for source_id, target_id, and type columns
- Given the migrations run in a fresh environment, they complete without errors and are idempotent (safe to re-run)
- Given the database is seeded, test data covers at least one project, several context_nodes with embeddings, and associated edges so downstream stories can run integration tests

## Technical Approach

This story introduces PostgreSQL with the `pgvector` extension as the persistence layer for the context graph, task execution, and project management domains. The implementation is migration-driven: a single migration creates all seven tables, enables pgvector, and builds all required indexes. A separate seed script populates development data. The existing SQLite-based orchestrator database is **not** affected — this is a new Postgres data layer for the managed project.

**Key decisions:**
- **Embedding dimension: 1536** (OpenAI ada-002 default; parameterizable via migration constant)
- **IVFFlat index** with `lists = 100` (appropriate for <1M rows; re-tune after data grows)
- **Migration tool:** `node-pg-migrate` (JS-based, programmatic runner, no ORM lock-in)
- **Connection:** `pg` pool via `DATABASE_URL` env var
- **UUID primary keys** (consistent with existing orchestrator schema pattern)

### Migration Plan

**Single migration file:** `001_core_schema.sql`

```sql
-- Enable pgvector
CREATE EXTENSION IF NOT EXISTS vector;

-- 7 tables + all indexes in dependency order:
-- 1. projects (no deps)
-- 2. context_nodes (depends on projects)
-- 3. context_edges (depends on context_nodes)
-- 4. node_locks (depends on context_nodes)
-- 5. tasks (depends on projects)
-- 6. task_results (depends on tasks)
-- 7. task_snapshots (depends on tasks)
```

All DDL uses `IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` for idempotency.

## Risk Assessment

### 1. IVFFlat Index Requires Training Data
**Risk:** IVFFlat indexes need sufficient rows to build effective clusters. With seed data of ~5 nodes, the index will exist but won't provide meaningful ANN performance.
**Mitigation:** Use `lists = 1` in the seed/dev migration. Document that production should re-create the index with `lists = sqrt(row_count)` once data exceeds 1,000 rows. Alternatively, consider HNSW index (no training needed) but ivfflat is specified in AC.

### 2. Embedding Dimension Lock-in
**Risk:** Hardcoding `vector(1536)` ties the schema to OpenAI ada-002 dimensions. Switching to a different model (e.g., 768-dim or 3072-dim) requires a migration.
**Mitigation:** This is acceptable for v1. Document the dimension in the migration comments. A future migration can `ALTER COLUMN` if needed.

### 3. Dual Database Complexity
**Risk:** The app now has two databases (SQLite for orchestrator, Postgres for context graph). Connection management, health checks, and error handling double.
**Mitigation:** Keep PG pool isolated in `pg-client.js`. Add PG connection to the existing health endpoint. Fail gracefully if PG is unavailable (orchestrator core functions remain on SQLite).

### 4. Seed Data Determinism
**Risk:** Seed embeddings must be valid 1536-dim vectors. Random floats work for schema validation but won't produce meaningful similarity results.
**Mitigation:** Use pre-computed embeddings from a known text set (e.g., embed the strings "hello world", "database schema", etc.) or use deterministic pseudo-random vectors seeded with a fixed value. Document that seed embeddings are for structural testing only.

### 5. Node Lock Expiry
**Risk:** No background process cleans up expired locks. Stale locks could block operations.
**Mitigation:** Add a `WHERE expires_at < now()` cleanup query to the lock-check endpoints. Document that a periodic cleanup job should be added in a future story.
