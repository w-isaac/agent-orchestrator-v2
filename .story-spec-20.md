# AOV-20: Auto task decomposition: complexity analysis, sub-task creation, and independent retry

As an orchestrator user, I want tasks to be automatically analysed for complexity and decomposed into sub-tasks so that large or multi-domain work can be parallelised across agents, tracked independently, and retried without disrupting sibling tasks. Complexity analysis considers token estimates and domain spread. Each sub-task inherits its own seed and budget from the parent. Results roll up to the parent on completion, and a failed sub-task can be retried (manually or via LLM-assisted strategy) while siblings continue uninterrupted.


## Acceptance Criteria
- Given a task is created or updated, when it exceeds the configured token or domain-count threshold, then the system automatically creates child sub-tasks with parent-child relationships recorded in the database.
- Given a task is decomposed, when sub-tasks are created, then each sub-task is assigned its own token budget and seed derived from the parent, and the parent's budget is reduced accordingly.
- Given all sub-tasks for a parent complete successfully, when the last sub-task transitions to 'done', then the parent task status rolls up to 'done' with aggregated token usage and output artifacts.
- Given a sub-task fails, when the failure is detected, then sibling sub-tasks continue processing unaffected and the failed sub-task is marked for retry.
- Given a sub-task is marked for retry, when manual retry is triggered or LLM-assisted retry strategy is selected, then the sub-task is re-queued with a fresh seed while preserving all sibling state.
- Given LLM-assisted decomposition is enabled, when a task is analysed, then the system calls the configured LLM to suggest decomposition boundaries and sub-task descriptions before creating child records.
- Given the task detail view, when a task has sub-tasks, then the UI displays the parent-child hierarchy, individual sub-task statuses, per-sub-task token budgets, and retry controls for failed sub-tasks.

## Screen Descriptions

### 1. Task Detail View (Modified)

**Layout:** Two-column on desktop (task metadata left, activity/output right). Single column on mobile/tablet.

**Key additions:**
- **Decomposition Status Banner** — appears below the task header when decomposition is in progress or complete. Shows: `Analysing complexity…` (loading) → `Decomposed into N sub-tasks` (done).
- **Sub-task Panel** — collapsible section below the main task body, always visible when sub-tasks exist. Contains the sub-task list table.
- **Parent Budget Summary** — inline chip next to the task token budget showing `Budget remaining: X / Original: Y` after budget has been distributed to sub-tasks.

**States:**

| State | UI |
|---|---|
| No sub-tasks | Sub-task panel hidden |
| Decomposition in progress | Banner with spinner: "Analysing task complexity…" |
| LLM suggesting boundaries | Banner: "LLM is proposing decomposition — preview below" + ghost sub-task rows |
| Sub-tasks created | Panel visible, each row shows status badge + budget |
| All done | Parent status auto-updates; banner: "All sub-tasks complete" |
| One or more failed | Failed rows highlighted in red; retry controls visible |

## ARCHITECTURE

Architecture spec complete. Key design decisions:

- **1 new table** (`sub_tasks`) with 18 columns, **4 new columns** on `stories`, **3 new settings keys** — all in a single migration `039_sub_tasks.sql`
- **6 new API endpoints** across 2 route files, **3 modified endpoints** on stories
- **4 new files** to create, **5 existing files** to modify
- **Zero new npm dependencies** — everything builds on existing `uuid`, `better-sqlite3`, `ws`, and `claude-runner.js`
- Sub-tasks are execution units within a stage, **not** full pipeline stories — keeps the scheduler changes contained
- LLM decomposition uses **sonnet** (classification task) via existing `runClaude()`
- Schema forward-compatible for nested decomposition via `parent_sub_task_id`, but v1 enforces depth = 1
