# AOV-21: Full conflict resolution: advisory locks, auto-merge, and re-queue for incompatible changes

As the orchestrator system, I need a complete conflict resolution pipeline so that concurrent agent dispatches do not corrupt shared state or produce contradictory changes. When multiple agents attempt to modify overlapping artifacts simultaneously, the system must prevent races via advisory locks, intelligently merge non-overlapping changes, produce a combined diff for compatible overlaps, and re-queue tasks whose changes are fundamentally incompatible. High-contention scenarios must be surfaced for observability.


## Acceptance Criteria
- Given a task is about to be dispatched, when the system checks node_locks, then an advisory lock with an expiry timestamp is acquired on all affected nodes before dispatch proceeds
- Given a lock exists on a node, when the expiry time has passed, then the lock is automatically released and the node becomes available for new locks
- Given two concurrent tasks modify non-overlapping sections of an artifact, when both complete, then changes are automatically merged without conflict and both diffs are applied
- Given two concurrent tasks modify overlapping but compatible sections, when both complete, then a single combined diff is produced and applied atomically
- Given two concurrent tasks produce fundamentally incompatible changes to the same section, when conflict is detected, then the lower-priority task is re-queued with conflict metadata attached
- Given a node or artifact experiences repeated lock contention beyond a configurable threshold, then a high-contention alert is emitted and recorded for analytics
- Given any lock acquisition, merge, or re-queue event occurs, then the event is logged with task ID, node ID, conflict type, and resolution outcome

## Screen Descriptions

### 1. Lock Status Dashboard (`/conflicts/locks`)

**Layout:** Full-width data view. Top: summary stat cards (row). Below: filterable table of active/expired locks.

**Key UI Elements:**
- **Stat cards (row of 4):** Active Locks | Expired Locks | Pending Expirations | Nodes Under Contention
- **Filter bar:** Node ID search, status filter (Active / Expired / Released), task ID search
- **`LockStatusTable`:** Columns — Node ID | Task ID | Acquired At | Expires At | Status | Actions
  - Status badge: `ACTIVE` (amber), `EXPIRED` (red), `RELEASED` (green)
  - Actions: "View Task", "Force Release" (destructive, requires confirm dialog)
- **Auto-refresh toggle** (30s interval default)

**States:**
- **Empty:** "No active locks" illustration + text
- **Loading:** Skeleton rows (5)
- **Error:** Inline error banner with retry button
- **Populated:** Table with pagination (25 rows/page)

## ARCHITECTURE

Architecture spec written to `/app/.arch-spec-output.md`. Here's a summary of what's covered:

**Data Model:** 4 new tables (`node_locks`, `conflict_records`, `conflict_events`, `contention_alerts`) + 2 new columns on `stories` across 4 migration files (039–042).

**API:** 12 endpoints under `/api/conflicts/*` spanning lock management, conflict resolution, task queue, analytics/alerts, and event audit log with CSV export. 3 new WebSocket event types via existing broadcaster.

**Services:** 4 new backend services — `lockManager` (TTL-based advisory locks), `mergeEngine` (three-way diff classification), `conflictDetector` (orchestration), `contentionMonitor` (threshold alerts).

**File Changes:** 12 new files, 4 modified. Scheduler integration acquires/releases locks around dispatch. Deployment agent feeds diffs into the conflict detector.

**Dependencies:** `diff3` + `csv-stringify` (backend), `react-diff-viewer-continued` + `recharts` (frontend).

**Risks:** 6 identified — lock starvation, race conditions in acquisition, incorrect merges, re-queue loops, migration compat, WebSocket flood — each with mitigations.

**Testing:** 3 unit test suites, 1 integration suite, 15 key scenarios covering concurrency safety, merge correctness, and observability.
