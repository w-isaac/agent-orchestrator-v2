# AOV-22: One-time v1 SQLite to v2 Postgres migration script

As a system operator, I need a one-time migration tool that reads the v1 SQLite database and imports all data into the v2 Postgres schema, so that existing project history and agent run data is preserved when cutting over to v2 without manual re-entry.


## Acceptance Criteria
- Given a valid v1 SQLite file path, when the migration runs, then all v1 projects are inserted into v2 with their original IDs preserved
- Given v1 stories, when migrated, then each story maps to a v2 task with status values translated to v2 equivalents
- Given v1 agent_runs, when migrated, then each run maps to a v2 task_result with all relevant fields populated
- Given v1 stages, when migrated, then the stages concept is dropped and no stage records appear in v2
- Given --dry-run flag is passed, when the migration runs, then it prints a summary of what would be created without writing to Postgres
- Given the migration is run a second time against the same v1 data, then no duplicate records are created in v2 (idempotent via upsert or existence checks)
- Given a migration failure on any record, then the error is logged with the source record ID and migration continues for remaining records, reporting a final success/failure count

# Architecture Spec: v1 SQLite â†’ v2 Postgres Migration Tool

## Technical Approach

**Standalone Node.js CLI script** (`src/scripts/migrate-v1.js`) that reads the existing v1 SQLite database and upserts records into a v2 Postgres database. The tool is independent of the running Express server â€” it imports no app code except shared constants.

**Key design decisions:**
- **Upsert via `ON CONFLICT`** â€” guarantees idempotency without pre-checking existence
- **Per-record error isolation** â€” each INSERT is its own try/catch; failures are collected, not thrown
- **Stream-style processing** â€” read all v1 records into memory (dataset is small: hundreds, not millions), then write sequentially per entity type
- **No ORM** â€” raw SQL via `better-sqlite3` (read) and `pg` (write) for full control over upsert semantics
- **Dry-run is offline** â€” `--dry-run` only opens SQLite, never connects to Postgres (answers Open Question #2)

**Entity mapping:**

| v1 Table | v2 Table | Notes |
|---|---|---|
| `projects` | `projects` | 1:1, preserve original UUIDs |
| `stories` | `tasks` | Status values translated via `StatusMapper` |
| `agent_runs` | `task_results` | Column rename mapping; v2 foreign keys resolved |
| `pipeline_stages` | *(dropped)* | Counted and reported as "skipped" |

## Risk Assessment

### 1. V2 schema mismatch
**Risk:** The v2 Postgres schema may differ from assumed column names/types above.
**Mitigation:** The `PostgresWriter` should validate table existence and column names on connect (before writing). Fail fast with a clear error listing missing columns.

### 2. Orphaned foreign keys
**Risk:** `agent_runs` referencing a `story_id` or `project_id` that doesn't exist in v1 (data corruption in source). The v2 FK constraint will reject the insert.
**Mitigation:** Migrate in dependency order: projects â†’ tasks â†’ task_results. Log the specific FK violation and source record ID; continue migration.

### 3. Timestamp format differences
**Risk:** v1 stores timestamps as ISO 8601 TEXT strings. Postgres `TIMESTAMPTZ` requires valid timestamp input. Malformed v1 strings will cause insert failures.
**Mitigation:** Parse and validate timestamps in the reader layer. Replace unparseable values with `NULL` (if column is nullable) and log a warning.

### 4. Large agent_runs dataset
**Risk:** If `agent_runs` has thousands of records, per-record inserts will be slow.
**Mitigation:** Batch upserts in groups of 100 using `INSERT ... VALUES (...), (...) ON CONFLICT DO UPDATE`. Still isolate errors by falling back to per-record insert for any failed batch.

### 5. Concurrent execution
**Risk:** Two operators run the migration simultaneously, causing race conditions on upserts.
**Mitigation:** Upserts are inherently safe for concurrent execution (Postgres `ON CONFLICT` is atomic). The final summary counts may be misleading but data integrity is preserved. Document in `--help`: "Safe to re-run; not designed for concurrent execution."

### E2E / manual validation

- Operator runs `node src/scripts/migrate-v1.js --sqlite-path ./data/orchestrator.db --pg-url $PG_URL --dry-run` against real v1 data and verifies counts match expected
- Operator runs live migration against a staging Postgres, then queries v2 tables to spot-check record integrity
- Operator re-runs to confirm idempotency (exit 0, 0 new records)
