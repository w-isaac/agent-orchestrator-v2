# AOV-23: Frontend scaffold: extract v1 static assets and serve via Docker

As a developer setting up the v2 project, I need the v1 frontend static files (HTML shell, CSS, shared JS utilities, and assets) extracted into the v2 project under client/, with the Express server configured to serve them and Docker Compose exposing them on the deploy port. This provides a working frontend shell without any API integration — just the static structure that loads cleanly in a browser, serving as the foundation for future v2 API integration work.


## Acceptance Criteria
- Given the v2 repo, when I inspect client/, then all base HTML/CSS/asset files from v1 are present
- Given Docker Compose is running, when I open the app in a browser, then the frontend shell loads with no 404s or console errors
- Given the frontend shell, when I inspect the page, then it renders the app layout with placeholder content


## Project Infrastructure Settings

- **deploy_port**: 3004 (use this as the host port in docker-compose.yml: `3004:3000`)
- **container_port**: 3000

## Technical Approach

This story is a **static asset scaffold** — no database, no new API endpoints, no new npm dependencies. The work is:

1. **Create** the `client/` source directory structure with v1 static assets (HTML shell, CSS, JS utilities, asset placeholders)
2. **Modify** the Express server (`src/server.js`) to serve the new `client/` directory directly (in addition to or replacing the existing `client/dist` path)
3. **Create** `docker-compose.yml` and `Dockerfile` to containerize and expose the app on a deploy port

The existing server already has `express.static` wired up for `client/dist/`. The decision point is whether to:
- **(A)** Place new static files directly in `client/dist/` (matching existing serve path), or
- **(B)** Restructure to serve from `client/` with a build step that copies to `dist/`

**Recommendation: Option A** — place hand-authored static files in `client/dist/` for zero-config compatibility with the existing server. The current `client/dist/` contains Vite-built React bundles; this story replaces them with the v1 static shell. When React is reintroduced later, the build pipeline overwrites `dist/` as before.

However, to keep source files separate from build artifacts (cleaner git history, clearer ownership), the better approach is to **add a parallel static-serve path**: place source files in `client/static/` and add a second `express.static` mount for development, while the production build copies them into `dist/`. This is slightly more complex but future-proof.

**Final decision: Place source files in `client/` root-level subdirectories (`client/css/`, `client/js/`, `client/assets/`) and update the Express static middleware to serve from `client/` directly when `client/dist/index.html` does not exist (dev/scaffold mode).** This matches the spec's file paths exactly and requires minimal server changes.

### Detailed `src/server.js` Change

**Current** (lines 79–84):
```js
const clientDist = join(__dirname, '../client/dist');
if (existsSync(join(clientDist, 'index.html'))) {
  app.use(express.static(clientDist));
  app.get('*', (_req, res) => res.sendFile(join(clientDist, 'index.html')));
}
```

**Proposed:**
```js
const clientDir = join(__dirname, '../client');
const clientDist = join(clientDir, 'dist');

if (existsSync(join(clientDist, 'index.html'))) {
  // Production: serve built React SPA
  app.use(express.static(clientDist));
  app.get('*', (_req, res) => res.sendFile(join(clientDist, 'index.html')));
} else if (existsSync(join(clientDir, 'index.html'))) {
  // Development/scaffold: serve static v1 shell
  app.use(express.static(clientDir));
  app.get('*', (_req, res) => res.sendFile(join(clientDir, 'index.html')));
}
```

This preserves backward compatibility: if a React build exists in `dist/`, it takes priority. Otherwise, the static scaffold in `client/` is served.

## Risk Assessment

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **v1 JS utilities contain API calls** — importing v1 `utils.js` verbatim may include `fetch()` calls to v1 endpoints, causing console errors and violating acceptance criteria | High | Audit every function in v1 `utils.js` before porting. Strip or stub any function that references API endpoints. Only port pure utility functions (formatters, DOM helpers). |
| 2 | **`client/dist/` override masks scaffold** — if `client/dist/index.html` exists (current state), the scaffold `client/index.html` is never served | Medium | Remove or rename `client/dist/` contents when deploying the scaffold. Document that `npm run build` will regenerate `dist/` and override the scaffold. |
| 3 | **Asset path collisions** — `client/css/` and `client/js/` paths could collide with future React build output in `client/dist/assets/` | Low | No collision: `dist/` is a separate directory. The `express.static` priority logic ensures clean separation. |
| 4 | **Docker port conflict** — port `8080` may conflict with other services on the developer's machine | Low | Make the host port configurable via `.env` file (`HOST_PORT=8080`). Document in `docker-compose.yml` comments. |
