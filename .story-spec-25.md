# AOV-25: Task board with v2 status model

As a user, I want a task board in the frontend that displays tasks using the v2 status-based model (queued, in_progress, complete) so I can monitor and manage tasks through their lifecycle. The board should fetch data from v2 API endpoints and support drag-and-drop to transition task status, replacing any v1 stage-based pipeline concepts.


## Acceptance Criteria
- Given the frontend is loaded, when I open the task board, then tasks are displayed grouped by v2 status: queued, in_progress, and complete
- Given the task board, when I view a task card, then it shows data fetched from v2 API endpoints (not v1 pipeline stages)
- Given the task board, when I drag a task card to a different status column, then the task status updates via v2 API and the UI reflects the change
- Given the task board, when I create a new task via the board UI, then it is submitted to the v2 task creation endpoint and appears in the queued column
- Given the task board, when an API call fails during a status transition, then the card reverts to its previous column and an error message is shown
- Given the task board is open, when tasks are updated by background processes, then the board reflects current status without requiring a full page reload

# Architecture Spec — Task Board with v2 Status Model

## Technical Approach

**Core insight:** The existing `stories` table already tracks status as `queued | in_progress | done`. Rather than creating a new table, the v2 task board is a **simplified view layer** over existing stories, exposed through new `/api/v2/tasks` endpoints that:

1. Map the internal `done` status to `complete` in API responses
2. Strip pipeline-stage complexity (no `current_stage_id`, `pipeline_gate_results`, etc.)
3. Support direct status transitions via `PATCH` (bypassing the stage-based `advance`/`retreat` flow)

**Frontend:** No React source currently exists in the repo (only pre-built `client/dist/`). New components will be created in `client/src/` with a Vite + React + TypeScript scaffold. The existing WebSocket infrastructure (`/ws`) will be reused for real-time updates.

**Status mapping:**

| Internal (DB) | v2 API | Board Column |
|---|---|---|
| `queued` | `queued` | Queued |
| `in_progress` | `in_progress` | In Progress |
| `done` | `complete` | Complete |

---

## Data Model Changes

**No schema migrations required.** The existing `stories` table already has the necessary fields:

| Column | Type | Usage in Task Board |
|---|---|---|
| `id` | TEXT (UUID) | Task identifier |
| `title` | TEXT | Card title |
| `description` | TEXT | Card description excerpt |
| `status` | TEXT | Column grouping (`queued`, `in_progress`, `done`) |
| `sort_order` | INTEGER | Card ordering within column |
| `project_id` | TEXT | Scope filtering |
| `display_id` | INTEGER | Human-readable ID shown on card |
| `created_at` | DATETIME | Sort fallback |
| `updated_at` | DATETIME | Change tracking |

**Indexes already in place:** Stories are queried by `status` and `project_id` extensively throughout the codebase. No new indexes needed.

**No new migrations.** The v2 API is a presentation layer over existing data.

## API Changes

### New route file: `/api/v2/tasks`

All endpoints scoped to a project via query parameter or request body.

#### `GET /api/v2/tasks?project_id={id}`

Fetch all tasks grouped by v2 status.

**Response:**
```json
{
  "tasks": [
    {
      "id": "uuid",
      "display_id": 12,
      "title": "Implement login",
      "description": "OAuth2 flow...",
      "status": "queued",          // mapped: done → complete
      "sort_order": 0,
      "project_id": "uuid",
      "created_at": "2026-04-15T...",
      "updated_at": "2026-04-15T..."
    }
  ]
}
```

#### `POST /api/v2/tasks`

Create a new task (story) with initial status `queued`.

**Request body:**
```json
{
  "project_id": "uuid",
  "title": "Task title",        // required
  "description": "Optional description"
}
```

**Response:** `201` with created task object (same shape as GET).

**Behavior:**
- Assigns to the project's Backlog stage (via `current_stage_id`) for internal pipeline compatibility
- Sets `status = 'queued'`, `sort_order = 0`
- Auto-generates `display_id` (sequential per project)
- Broadcasts `task_created` via WebSocket

#### `GET /api/v2/tasks/:id`

Single task detail.

**Response:** Task object (same shape as list item).

#### `PATCH /api/v2/tasks/:id`

Update task fields — primarily used for drag-and-drop status transitions.

**Request body (any combination):**
```json
{
  "status": "in_progress",       // accepts: queued, in_progress, complete
  "title": "Updated title",
  "description": "Updated desc",
  "sort_order": 3
}
```

**Response:** Updated task object.

**Behavior:**
- Maps incoming `complete` → internal `done` before DB write
- Broadcasts `task_updated` via WebSocket
- Returns `400` if status value is invalid
- Returns `404` if task not found

#### `POST /api/v2/tasks/reorder`

Bulk reorder tasks within a status column.

**Request body:**
```json
{
  "task_ids": ["uuid1", "uuid2", "uuid3"]
}
```

**Behavior:** Sets sequential `sort_order` values. Reuses existing `stories/reorder` logic.
