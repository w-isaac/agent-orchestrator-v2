# AOV-28: Force-directed graph: read-only D3 rendering with zoom/pan and typed nodes

As a user viewing the agent orchestrator dashboard, I need a read-only force-directed graph visualization built with D3.js that renders artifacts, tasks, and context as typed nodes connected by weighted edges. The visualization must support smooth zoom and pan navigation, visually distinguish node types by color/shape, and display edge weights on hover. No interaction (add/edit/delete) is in scope — this story covers rendering and navigation only.


## Acceptance Criteria
- Given context graph data is available, when the visualization loads, then all nodes and edges render using a D3 force-directed layout
- Given the graph is rendered, when the user zooms or pans, then the viewport updates smoothly without layout recalculation
- Given nodes of different types (artifact, task, context), when rendered, then each type is visually distinct via color and/or shape
- Given edges with weights, when the user hovers over an edge, then the weight value is displayed in a tooltip or label
- Given the visualization is read-only, when the user attempts to add, edit, or delete nodes/edges, then no such interaction is possible

# Architecture Spec: Force-Directed Graph Visualization

## Technical Approach

Add a read-only D3.js force-directed graph to the Agent Orchestrator dashboard. The graph renders **existing domain entities** — artifacts, tasks (stories), and context — as typed nodes, with relationships (story dependencies, artifact-to-story links) as weighted edges.

**Strategy:**
- **Backend:** New REST endpoint assembles graph data by querying existing tables (`stories`, `artifacts`, `story_dependencies`) and a new `context_graph_edges` table for explicit weighted edges. Returns `{ nodes, edges }` JSON.
- **Frontend:** New React component tree (`ForceGraph` + children) renders SVG via D3. React owns the container/toolbar/legend/tooltip; D3 owns the SVG internals (force simulation, node/edge rendering). Zoom/pan via `d3-zoom`.
- **No WebSocket:** Graph loads on mount via REST fetch. Real-time updates are out of scope per the story.
- **Database:** SQLite (better-sqlite3), consistent with existing stack. One new table for explicit graph edges; node data is derived from existing tables.

---

## Data Model Changes

### New Table: `context_graph_edges`

Stores explicit weighted edges between graph entities that aren't captured by existing foreign keys.

```sql
-- Migration: 039_context_graph.sql

CREATE TABLE IF NOT EXISTS context_graph_edges (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL REFERENCES projects(id),
  source_type TEXT NOT NULL CHECK(source_type IN ('artifact','task','context')),
  source_id   TEXT NOT NULL,
  target_type TEXT NOT NULL CHECK(target_type IN ('artifact','task','context')),
  target_id   TEXT NOT NULL,
  weight      REAL NOT NULL DEFAULT 1.0,
  created_at  TEXT NOT NULL,
  UNIQUE(project_id, source_type, source_id, target_type, target_id)
);

CREATE INDEX IF NOT EXISTS idx_cge_project ON context_graph_edges(project_id);
CREATE INDEX IF NOT EXISTS idx_cge_source  ON context_graph_edges(source_type, source_id);
CREATE INDEX IF NOT EXISTS idx_cge_target  ON context_graph_edges(target_type, target_id);
```

### Node Derivation (no schema changes needed)

| Node Type | Source Table | ID | Label |
|---|---|---|---|
| `task` | `stories` | `stories.id` | `stories.title` |
| `artifact` | `artifacts` | `artifacts.id` | `artifacts.title` |
| `context` | `context_graph_edges` (unique source/target where type=`context`) | synthetic | from edge metadata |

> **Note:** "Context" nodes don't have a dedicated table. They can be seeded via the `context_graph_edges` table or derived from `stories.prompt_context` (migration 022). For MVP, context nodes are represented as entries referenced in `context_graph_edges` with `type='context'`.

### Implicit Edges (derived, not stored)

| Relationship | Source | Target | Weight Logic |
|---|---|---|---|
| Story → Artifact | `stories.id` | `artifacts.story_id` | `1.0` (direct ownership) |
| Story → Story dependency | `story_dependencies.story_id` | `story_dependencies.depends_on` | `0.8` (dependency link) |

## API Changes

### `GET /api/graph/:projectId`

Returns the assembled graph for a project.

**Response shape:**
```json
{
  "nodes": [
    { "id": "story-abc123", "type": "task", "label": "Add login flow" },
    { "id": "artifact-def456", "type": "artifact", "label": "auth-spec.md" },
    { "id": "context-ghi789", "type": "context", "label": "Auth requirements" }
  ],
  "edges": [
    { "source": "story-abc123", "target": "artifact-def456", "weight": 1.0 },
    { "source": "story-abc123", "target": "context-ghi789", "weight": 0.65 }
  ]
}
```

**Query logic:**
1. Fetch all non-done stories for the project → `task` nodes
2. Fetch all non-superseded artifacts for those stories → `artifact` nodes
3. Fetch all `context_graph_edges` for the project → explicit edges + `context` nodes
4. Generate implicit edges (story→artifact, story→story dependency)
5. Deduplicate and merge edges; prefer explicit weight over implicit
