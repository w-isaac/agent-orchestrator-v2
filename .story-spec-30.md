# AOV-30: Context preview panel: display retrieved artifacts before dispatch

As an operator reviewing a task before dispatch, I want to open a read-only preview panel that lists all context artifacts that will be sent to the agent, so that I can understand what context the agent will receive and assess the total token footprint before committing to dispatch.


## Acceptance Criteria
- Given a task ready for dispatch, when I open the preview panel, then all retrieved context artifacts are listed
- Given each artifact in the preview, then its title, type, token count, and relevance score are displayed
- Given the preview panel, then a summary shows the total token count across all artifacts
- Given the preview panel, then it is read-only with no budget controls or filtering actions

## Technical Approach

Add a read-only slide-over panel to the Task Dispatch view that fetches and displays all context artifacts for a story before agent dispatch. The implementation spans three layers:

1. **Database** — Extend the `artifacts` table with `token_count` (INTEGER) and `relevance_score` (REAL) columns so agents can record token footprint and relevance at artifact creation time.
2. **API** — New `GET /api/stories/:id/context-preview` endpoint that returns non-superseded artifacts with token/relevance metadata and a computed summary (total count, total tokens).
3. **Frontend** — New `ContextPreviewPanel` component tree mounted in the existing task dispatch view, using lazy fetch on panel open.

Key design decisions:
- **Lazy fetch** on panel open (not eager) — keeps the loading skeleton visible per the UX spec and avoids unnecessary queries for stories the operator never previews.
- **Read-only** — no mutations, no filtering controls. Pure display.
- **Token count computed at artifact creation** — agents already produce content; token counting is added to the artifact creation flow rather than computed on-the-fly at preview time.

## Data Model Changes

### Migration 039: `artifacts_context_preview_columns.sql`

```sql
-- Add token_count and relevance_score to artifacts for context preview
ALTER TABLE artifacts ADD COLUMN token_count     INTEGER DEFAULT 0;
ALTER TABLE artifacts ADD COLUMN relevance_score REAL    DEFAULT NULL;

CREATE INDEX IF NOT EXISTS idx_artifacts_story_active
  ON artifacts(story_id, superseded)
  WHERE superseded = 0;
```

| Column | Type | Nullable | Default | Purpose |
|---|---|---|---|---|
| `token_count` | INTEGER | No | `0` | Token footprint of artifact content |
| `relevance_score` | REAL | Yes | `NULL` | 0.00–1.00 relevance to dispatch context |

**No new tables required.** The existing `artifacts` table already stores per-story, per-stage artifacts with content and type — only the two metadata columns are missing.

The partial index on `(story_id, superseded) WHERE superseded = 0` optimizes the context-preview query, which only fetches active (non-superseded) artifacts.

## API Changes

### New Endpoint

**`GET /api/stories/:id/context-preview`**

Returns all active (non-superseded) artifacts for a story, with a computed summary.

**Response `200`:**
```json
{
  "artifacts": [
    {
      "id": "uuid",
      "title": "Design Spec v2",
      "type": "design_doc",
      "token_count": 3420,
      "relevance_score": 0.92,
      "stage_id": "uuid",
      "created_at": "2026-04-15T10:00:00Z"
    }
  ],
  "summary": {
    "artifact_count": 5,
    "total_tokens": 12340
  }
}
```

**Response `404`:** `{ "error": "Story not found" }`

**Query (SQLite):**
```sql
SELECT id, title, type, token_count, relevance_score, stage_id, created_at
FROM artifacts
WHERE story_id = ? AND superseded = 0
ORDER BY relevance_score DESC, created_at ASC
```

Summary is computed in-app: `artifact_count = artifacts.length`, `total_tokens = SUM(token_count)`.

### Modified Endpoint

**`POST /api/stories/:id/artifacts`** — Accept two new optional fields in the request body:

| Field | Type | Required | Default |
|---|---|---|---|
| `token_count` | integer | No | `0` |
| `relevance_score` | number | No | `null` |

The INSERT statement is extended to include these columns. No other behavior changes.
