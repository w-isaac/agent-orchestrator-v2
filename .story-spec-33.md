# AOV-33: Pipeline view page with stage columns, story cards, and detail panel

As a user, I want to view the pipeline as a set of stage columns so that I can understand the current state of all stories across each stage at a glance. The pipeline view should fetch stage and story data from GET /api/v2/pipelines, render each stage as a column with a header, and display story cards within each column. Each card should show the story's title, a status badge, and the assignee. Clicking a story card should open a detail panel without navigating away. A navigation control should allow returning to the dashboard.


## Acceptance Criteria
- Given the pipeline view loads, when data is fetched from GET /api/v2/pipelines, then pipeline stages are rendered as distinct columns with stage name headers
- Given each story card in a stage column, then it displays the story title, a colour-coded status badge, and the assignee
- Given the pipeline view, when I click a story card, then a detail panel opens showing full story details without leaving the page
- Given the pipeline view, when the API returns an error or is loading, then appropriate loading and error states are displayed to the user
- Given the pipeline view, when I click the back/navigation control, then I am returned to the project dashboard
- Given the pipeline view, when stages contain no stories, then empty columns are still rendered with their stage headers visible

# Architecture Spec: Pipeline View Page

**Story:** Pipeline view page with stage columns, story cards, and detail panel

---

## Technical Approach

Implement a kanban-style pipeline view as a new React page in the existing Agent Orchestrator client. The view leverages the **existing** `GET /api/projects/:id/pipeline` endpoint which already returns stages with nested stories — no backend changes required. The detail panel fetches enriched story data from the existing `GET /api/stories/:id` endpoint on card click.

**Key decisions:**
- **No new API endpoints** — existing `/api/projects/:id/pipeline` already returns the exact shape needed (stages → stories with status, assignee data)
- **No migrations** — `pipeline_stages` and `stories` tables already contain all required fields (title, status, current_stage_id, created_at, updated_at, description, acceptance_criteria)
- **Client-only feature** — 6 new React components + 1 modified component
- **Route:** `/projects/:id/pipeline` (project-scoped, consistent with existing `GET /api/projects/:id/pipeline`)

---

## Data Model Changes

**None.** The existing schema fully supports this feature:

| Table | Relevant Columns | Usage |
|---|---|---|
| `pipeline_stages` | `id`, `name`, `icon`, `stage_order`, `project_id` | Stage column headers + ordering |
| `stories` | `id`, `title`, `status`, `current_stage_id`, `description`, `acceptance_criteria`, `created_at`, `updated_at`, `display_id`, `priority`, `epic` | Card display + detail panel |
| `agent_runs` | `agent_role`, `story_id` | Assignee/agent info on cards |

**Note on "assignee":** The UX spec references an assignee with name + avatar. In the Agent Orchestrator domain, the "assignee" is the **agent role** (e.g., `engineering`, `qa`, `design`). The `agent_roster` table provides `name`, `icon`, and `description` per role. The most recent `agent_runs.agent_role` for a story serves as the current assignee. The `agent_roster.icon` (emoji) serves as the avatar.
