# AOV-35: TaskDispatcher service: dispatch-collect-validate lifecycle with unit tests

As a developer building the agent orchestration pipeline, I need a taskDispatcher service that manages the full task lifecycle so that tasks are reliably submitted to the ClaudeCodeAdapter, polled until completion, and their results validated before being stored. The dispatcher is the central coordinator between incoming task requests and the underlying adapter, ensuring errors are captured and output integrity is enforced at each stage.


## Acceptance Criteria
- Given a task submission, when the dispatcher receives it, then it calls ClaudeCodeAdapter.submit with the correct task payload
- Given a running task, when the dispatcher polls checkStatus, then it continues polling until the task reaches a terminal state (complete or failed)
- Given a completed task, when results are collected, then the output is validated before being stored — invalid output must not be persisted
- Given a failed task, when the adapter reports failure, then the dispatcher records the error with sufficient detail for diagnosis and retry

# Architecture Spec: TaskDispatcher Service — Dispatch-Collect-Validate Lifecycle

## Technical Approach

Introduce a **TaskDispatcher** service layer that owns the full task lifecycle: submit → poll → collect → validate → store. It sits between the API layer and the existing `ClaudeCodeAdapter` (implemented as `runClaude` in `src/agents/claude-runner.js`).

**Key design decisions:**

- **New `tasks` table** — decoupled from `agent_runs`; tasks are first-class entities with their own status machine, validation state, and retry lineage
- **Service-layer polling** — the dispatcher polls `runClaude` completion internally (server-side); the frontend polls the `/api/tasks` endpoint for UI updates (5s interval)
- **Validation gate** — output passes through a pluggable validator before persistence; invalid output is kept in-memory on the task record (marked `invalid`) but never written to `artifacts`
- **Retry via new task** — retries create a new task linked to the original via `source_task_id`, preserving full audit trail
- **SQLite** — consistent with existing stack (better-sqlite3), not PostgreSQL

**Status state machine:**
```
queued → dispatched → running → validating → complete
                                           → invalid
                    → failed
```

### Migration file: `039_tasks_table.sql`

```sql
CREATE TABLE IF NOT EXISTS tasks (
  id                TEXT PRIMARY KEY,
  type              TEXT NOT NULL,
  payload           TEXT NOT NULL,
  priority          TEXT NOT NULL DEFAULT 'normal',
  timeout_seconds   INTEGER,
  status            TEXT NOT NULL DEFAULT 'queued',
  output            TEXT,
  validation_error  TEXT,
  error_code        TEXT,
  error_message     TEXT,
  error_stack       TEXT,
  adapter_response  TEXT,
  source_task_id    TEXT REFERENCES tasks(id),
  retry_count       INTEGER NOT NULL DEFAULT 0,
  last_polled_at    TEXT,
  submitted_by      TEXT,
  created_at        TEXT NOT NULL,
  updated_at        TEXT NOT NULL,
  dispatched_at     TEXT,
  completed_at      TEXT
);

CREATE INDEX IF NOT EXISTS idx_tasks_status ON tasks(status);
CREATE INDEX IF NOT EXISTS idx_tasks_type ON tasks(type);
CREATE INDEX IF NOT EXISTS idx_tasks_source ON tasks(source_task_id);
CREATE INDEX IF NOT EXISTS idx_tasks_created ON tasks(created_at);
```
