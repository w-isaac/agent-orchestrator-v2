# AOV-36: Task data model, migrations, and CRUD endpoints

As a backend developer, I need the core task-related database tables and CRUD API endpoints so that the rest of the system can create, track, and manage tasks through their lifecycle. This includes a single migration file (039) that creates the tasks, task_seed_nodes, preflight_checks, task_locks, and task_lifecycle_events tables, along with four REST endpoints for task management. Context_nodes and context_edges are explicitly excluded as they were delivered in AOV-28.


## Acceptance Criteria
- Given migration 039 is run, a 'tasks' table exists with columns: id, project_id, title, description, status, priority, budget, created_at, updated_at
- Given migration 039 is run, a 'task_seed_nodes' junction table exists linking task IDs to context node IDs
- Given migration 039 is run, a 'preflight_checks' table exists with task_id FK, check_type, status, detail, and timestamps
- Given migration 039 is run, a 'task_locks' table exists with task_id FK, resource_key, acquired_at, and released_at
- Given migration 039 is run, a 'task_lifecycle_events' table exists with task_id FK, status, timestamp, and a JSON payload column
- When POST /api/tasks is called with valid task data and seed node IDs, then a task is created with its seed nodes and a 201 response is returned
- When GET /api/tasks is called with optional status and/or priority query params, then a filtered list of tasks is returned
- When GET /api/tasks/:id is called with a valid task ID, then the task is returned including its associated seed nodes and lifecycle events
- When PATCH /api/tasks/:id is called with mutable fields (title, description, status, priority, budget), then the task is updated and the updated record is returned
- Given unit tests exist for all 4 endpoints and the migration schema, when the test suite runs, all tests pass

# Architecture Specification: Task Data Model, Migrations & CRUD Endpoints

## Technical Approach

Add the core task management domain to the Agent Orchestrator. This involves:

1. **Single migration file** (`039_tasks.sql`) creating 5 new tables for the task lifecycle
2. **New Express router** (`src/api/tasks.js`) with 4 CRUD endpoints following existing patterns (UUID IDs, ISO timestamps, `getDb()` singleton, `broadcast()` for WebSocket events)
3. **New React components** (6 components + 2 page-level wrappers) in `src/components/tasks/` for the Task List, Task Detail, and Create Task flows

The implementation follows established conventions:
- SQLite via `better-sqlite3` with WAL mode and foreign keys
- Sequential numbered migration files in `src/db/migrations/`
- Express routers using `getDb()`, `uuidv4()`, and `broadcast()` from existing modules
- Vitest + supertest for API tests with in-memory SQLite
- Pre-built React SPA served from `client/dist/`
