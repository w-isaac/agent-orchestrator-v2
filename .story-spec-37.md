# AOV-37: Pre-flight validation and budget estimation service

As a task dispatcher, I need a pre-flight check service that validates a task is safe and feasible to dispatch before committing resources. The service runs four sequential checks — freshness, locks, budget, and failure pattern — and short-circuits on the first failure. I also need a budget estimation endpoint so callers can forecast token costs before dispatching. This ensures tasks are only dispatched when all conditions are met, preventing wasted agent runs and budget overruns.


## Acceptance Criteria
- Given a task ID, when POST /api/tasks/:id/preflight is called, then the service runs freshness, locks, budget, and failure_pattern checks in sequence and returns an array of check results
- Given all four checks pass, when pre-flight completes, then each check is recorded in the preflight_checks table with status 'pass' and detail, and the task status transitions to 'dispatched'
- Given a check fails, when pre-flight encounters the failure, then remaining checks are skipped (short-circuit), the failed check is recorded in preflight_checks with status 'fail' and detail, and the task status transitions to 'pre_flight_failed'
- Given a task initiates pre-flight, when the process begins, then the task status transitions to 'pre_flight'
- Given seed_node_ids, when POST /api/tasks/estimate-budget is called, then the endpoint returns an estimated_tokens value calculated from the content sizes of the provided seed nodes
- Given varying context sizes, when budget estimation is requested, then estimated_tokens scales proportionally with total seed node content size
- Given a pre-flight pass scenario, when unit tests run, then all four checks return pass and the full results array is returned
- Given a pre-flight fail scenario, when unit tests run, then the first failing check causes short-circuit and only checks up to and including the failure are in the results
- Given a budget estimation scenario with varying context sizes, when unit tests run, then estimated_tokens reflects the correct calculation for each input

# Architecture Specification: Pre-flight Validation & Budget Estimation Service

## Technical Approach

Add a pre-flight validation pipeline and budget estimation service to the existing story dispatch flow. The spec uses "task" terminology — in this codebase, the equivalent entity is **stories**. All endpoints are mounted on the existing stories router.

**Key design decisions:**

- **Synchronous request/response** — Pre-flight checks run inline within a single HTTP request (all four checks are fast DB/memory operations, not external calls). No polling or WebSocket needed for the check itself; the WebSocket broadcast fires once on completion for UI reactivity.
- **Short-circuit evaluation** — Checks execute in a fixed sequence (`freshness → locks → budget → failure_pattern`). On first failure, remaining checks are recorded as `skipped` and the loop exits.
- **Transaction-wrapped** — All check results and the story status transition happen in a single SQLite transaction, ensuring atomicity.
- **Service module** — Business logic lives in a new `src/services/preflightService.js`, keeping the route handler thin. Each check is a pure function receiving `(db, story)` and returning `{ status, detail }`.
- **Budget estimation** — Stateless computation. Accepts seed node IDs, looks up content sizes from the `artifacts` table, and returns a token estimate using a linear scaling formula.

## Data Model Changes

### New Table: `preflight_checks`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | TEXT | PRIMARY KEY | UUID |
| `story_id` | TEXT | NOT NULL, FK → stories(id) | Story being validated |
| `run_id` | TEXT | NOT NULL | Groups checks from a single pre-flight invocation |

| `check_order` | INTEGER | NOT NULL | 1–4, execution sequence |
| `status` | TEXT | NOT NULL | `pass`, `fail`, `skipped` |
| `detail` | TEXT | | Human-readable result detail |
| `created_at` | TEXT | NOT NULL | ISO-8601 timestamp |

**Indexes:**
- `idx_preflight_checks_story_id` on `(story_id)`
- `idx_preflight_checks_run_id` on `(run_id)`

### Existing Table Changes

**`stories`** — No schema change needed. The `status` column is a free-form TEXT field. We add two new status values (`pre_flight`, `pre_flight_failed`) as application-level constants. The existing `pre_check_status` and `pre_check_meta` columns (migration 033) can be repurposed to track the latest pre-flight outcome for quick reads without joining.

CREATE TABLE IF NOT EXISTS preflight_checks (
  id          TEXT PRIMARY KEY,
  story_id    TEXT NOT NULL REFERENCES stories(id),
  run_id      TEXT NOT NULL,
  check_name  TEXT NOT NULL,
  check_order INTEGER NOT NULL,
  status      TEXT NOT NULL,
  detail      TEXT,
  created_at  TEXT NOT NULL

CREATE INDEX idx_preflight_checks_story_id ON preflight_checks(story_id);
CREATE INDEX idx_preflight_checks_run_id ON preflight_checks(run_id);

## API Changes

### 1. `POST /api/stories/:id/preflight`

Run the four-check pre-flight validation sequence.

**Request:** No body required.

**Response (200):**
```json
{
  "run_id": "uuid",
  "story_id": "uuid",
  "status": "pass" | "fail",
  "checks": [
    { "check_name": "freshness",       "status": "pass", "detail": "Story updated 2m ago" },
    { "check_name": "locks",           "status": "pass", "detail": "No active locks" },
    { "check_name": "budget",          "status": "fail", "detail": "Estimated 150k tokens exceeds 100k limit" },
    { "check_name": "failure_pattern", "status": "skipped", "detail": null }
  ]
}
```

**Error responses:**
- `404` — Story not found
- `409` — Story already in `pre_flight` or `dispatched` status

**Side effects:**
- Sets `stories.status` to `pre_flight` at start of request
