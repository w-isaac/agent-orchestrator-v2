# AOV-39: Complete task lifecycle: dispatch, collect, graph update, and 7 WebSocket events

As a developer building on the agent orchestrator, I need the full task lifecycle to be implemented end-to-end so that tasks can be dispatched to agents, results collected and validated, and the context graph updated with outputs — all with real-time WebSocket visibility. Currently the system can create and pre-flight tasks but cannot complete the delivery loop. This story closes that gap by implementing dispatch (lock + status transition + WS event), collect (receive raw result + normalize + validate), applyResults (write nodes/edges to context graph + transition to graph_updated), failure handling (status → failed + lock release + lifecycle event), all 7 WebSocket event types, and integration tests covering both the happy path (create → graph_updated) and the failure recovery path (dispatch fails → failed).


## Acceptance Criteria
- Given a task in preflight status, when taskLifecycle.dispatch(taskId) is called, then it acquires task_lock, transitions status to dispatched, and emits a task_dispatched WebSocket event
- Given a dispatched task, when taskLifecycle.collect(taskId, rawResult) is called, then it runs the normalizer and validator, transitions status through collecting to validated, and emits task_collected and task_validated WebSocket events
- Given a validated task, when taskLifecycle.applyResults(taskId) is called, then new nodes and edges are written to the context graph and status transitions to graph_updated with a task_graph_updated WebSocket event emitted
- Given any stage fails, when the failure is caught, then status transitions to failed, the lifecycle event is logged with payload, and the task_lock is released
- Given the system is running, all 7 WebSocket event types are broadcast: task_created, task_preflight_update, task_status_changed, task_dispatched, task_collected, task_validated, task_graph_updated
- All status transitions write a row to task_lifecycle_events table with timestamp and payload
- Integration test passes covering the full lifecycle from task create through graph_updated status
- Integration test passes covering the failure recovery path where dispatch fails and task is marked failed with lock released

## Screen Descriptions

### 1. Task Detail View (`/tasks/:taskId`)

**Layout:** Single-column content area with a sticky header bar. Three main sections stacked vertically: Status Header, Lifecycle Timeline, and Action Panel.

**Key UI Elements:**
- **Status Header** — task ID (monospace), current status badge (color-coded pill), lock indicator icon (locked/unlocked), last-updated timestamp
- **Lifecycle Timeline** — vertical stepper showing all status stages in order: `created → preflight → dispatched → collecting → validated → graph_updated` (or `→ failed`). Each step shows: status label, timestamp (when reached), payload excerpt (collapsible)
- **Action Panel** — context-sensitive buttons: "Dispatch" (visible when status = `preflight`), "Re-dispatch" (visible when status = `failed`), "View Graph" (visible when status = `graph_updated`)
- **Lifecycle Events Table** — paginated table below the timeline: columns `Event`, `Status`, `Timestamp`, `Payload (expandable)`

**States:**
| State | Behavior |
|---|---|
| Empty | Skeleton loaders for timeline steps; "Loading task…" label |
| Loading | Shimmer on timeline steps and status badge |
| Error | Inline error banner: "Could not load task. [Retry]" |
| `preflight` | "Dispatch" button enabled; steps 1–2 filled, 3–6 greyed |
| `dispatched` | Step 3 active/pulsing; lock icon shown as locked |
| `collecting` | Step 4 active/pulsing |
| `validated` | Steps 1–5 filled; "Apply Results" button shown |
| `graph_updated` | All steps filled green; "View Graph" CTA |
| `failed` | Failed step highlighted red; failure payload shown; lock released indicator |

## ARCHITECTURE

Architecture spec written to `/app/architecture.md`.

**Summary of what's specified:**

- **2 new tables:** `context_nodes` (with type, label, content, confidence, staleness_ttl_ms) and `context_edges` (source/target node FKs, relationship_type) — single migration `039_context_graph.sql`
- **3 lifecycle endpoints** on stories router: `POST /dispatch`, `POST /collect`, `POST /apply-results` — delegating to a `taskLifecycle` service with in-memory locking
- **4 context graph endpoints**: CRUD + BFS subgraph traversal at `/api/context-nodes`
- **7 WebSocket events** broadcast through existing `ws/broadcaster.js`, persisted to `system_events`
- **8 files to create** (migration, 3 services, 1 router, 3 test suites) + **3 files to modify** (server.js, broadcaster.js, stories.js)
- **9 frontend components** specified for future client scaffold (ContextNodeList, ContextNodeCard, NodeDetailPanel, EdgeList, SubgraphExplorer, NodeCreateForm, NormalizedResultPanel, ValidatorReport, ConfidenceGauge)
- **No new dependencies** — entire implementation uses existing stack
- **5 risks** addressed: lock loss on restart, dispatch race conditions, large graph writes, BFS depth explosion, backward compatibility of new statuses
