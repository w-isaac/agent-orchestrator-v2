# AOV-43: Auto-Pack button: greedy knapsack artifact selection within token budget

As a user of the context preview panel, I want an Auto-Pack button that automatically selects the best set of artifacts within the token budget so that I don't have to manually toggle artifacts to maximize context quality. The button triggers a greedy knapsack algorithm that sorts artifacts by their relevance_score/token_count ratio in descending order, then greedily selects artifacts one by one until adding the next would exceed the budget. After auto-packing, the toggle states and budget bar reflect the new selection.


## Acceptance Criteria
- Given the context preview panel is open, an Auto-Pack button is visible alongside the budget controls
- When the user clicks Auto-Pack, artifacts are sorted by relevance_score/token_count ratio in descending order
- When the user clicks Auto-Pack, artifacts are greedily selected in ratio order until adding the next artifact would exceed the token budget
- When auto-packing completes, all artifact toggle states update to reflect the selected set and the budget bar updates to show current usage
- Given an empty artifact list, auto-pack completes without error and no artifacts are selected
- Unit tests cover: empty artifacts, all artifacts fit within budget, no artifacts fit within budget, equal ratios, and various budget sizes

## Technical Approach

The Auto-Pack feature adds a client-side greedy knapsack algorithm that selects the optimal set of artifacts within a token budget. The implementation spans three layers:

1. **Database** — Extend the `artifacts` table with `token_count` (INTEGER) and `relevance_score` (REAL) columns so the algorithm has data to work with.
2. **API** — Add a dedicated `POST /api/stories/:id/artifacts/auto-pack` endpoint that accepts a `budget` parameter, runs the greedy knapsack server-side, and returns the selected artifact IDs. This answers Open Question #1 in favor of a **server-side algorithm** — while the UX spec suggests client-side, running it server-side keeps the door open for ML-based ranking and avoids shipping all artifact data to the client.
3. **Client** — New `AutoPackButton` component, modifications to `ContextPreviewPanel`, `BudgetBar`, and `ArtifactToggleRow` per the UX spec. The client calls the API endpoint and updates toggle state from the response.

**Open Question Resolutions:**
| # | Question | Decision |
|---|---|---|
| 1 | Algorithm location | Server-side — enables future ML ranking; client gets selected IDs only |
| 2 | Relevance score visibility | Hidden by default; shown as opt-in column (UX spec marks it optional) |
| 3 | Manual override after auto-pack | Users can freely toggle after auto-pack; next auto-pack resets all |
| 4 | Token budget source | System setting stored in `settings` table; overridable per-request via API param |
| 5 | Animation fidelity | All toggles flip at once (simpler; avoids latency perception) |

## Data Model Changes

### Migration: `039_add_autopack_fields_to_artifacts.sql`

```sql
-- Add token_count and relevance_score to artifacts for auto-pack algorithm
ALTER TABLE artifacts ADD COLUMN token_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE artifacts ADD COLUMN relevance_score REAL NOT NULL DEFAULT 0.0;

CREATE INDEX IF NOT EXISTS idx_artifacts_story_relevance
  ON artifacts(story_id, relevance_score DESC);
```

**Rationale:**
- `token_count` — Stores pre-computed token count for each artifact's content. Populated at artifact creation time.
- `relevance_score` — Float 0.0–1.0 representing artifact relevance. Set by upstream processes (agent runs, user input, or future ML).
- Index on `(story_id, relevance_score DESC)` optimizes the sorted fetch needed by the algorithm.

**No new tables required.** The existing `artifacts` table is sufficient; the `settings` table already exists for storing the default token budget.

| Column | Type | Default | Nullable | Purpose |
|---|---|---|---|---|
| `token_count` | INTEGER | 0 | NOT NULL | Pre-computed token count of artifact content |
| `relevance_score` | REAL | 0.0 | NOT NULL | Relevance ranking (0.0–1.0) |

## API Changes

### 1. `POST /api/stories/:id/artifacts/auto-pack` (NEW)

Runs the greedy knapsack algorithm server-side and returns selected artifact IDs.

**Request:**
```json
{
  "budget": 8000
}
```

| Param | Type | Required | Description |
|---|---|---|---|
| `budget` | integer | Yes | Token budget ceiling |

**Response (200):**
```json
{
  "selected_artifact_ids": ["uuid-1", "uuid-3", "uuid-7"],
  "total_tokens": 4200,
  "budget": 8000,
  "artifact_count": 3
}
```

**Error Responses:**
- `400` — Missing/invalid `budget` parameter
- `404` — Story not found

**Algorithm (pseudocode):**
```
1. Fetch non-superseded artifacts for story: WHERE story_id = :id AND superseded = 0
2. Compute ratio = relevance_score / token_count for each (skip if token_count = 0)
3. Sort by ratio DESC
4. Greedily select until next artifact would exceed budget
5. Return selected IDs + total tokens used
```
