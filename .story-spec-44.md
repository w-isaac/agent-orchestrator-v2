# AOV-44: Context graph: PATCH and DELETE node endpoints with cascade and WebSocket broadcasts

As a developer consuming the context graph API, I need PATCH and DELETE endpoints for graph nodes so that I can update node properties and remove nodes (with all their attached edges) through the existing graph router introduced in AOV-28. When a node is deleted, all edges connected to it must be cascade-deleted automatically. Both mutations must broadcast corresponding WebSocket events so connected clients receive real-time updates without polling.


## Acceptance Criteria
- Given a valid node ID, when PATCH /api/context-graph/nodes/:id is called with any subset of {label, type, properties, x, y, pinned}, then the node is updated and the updated node is returned with a 200 response
- Given a valid node ID, when DELETE /api/context-graph/nodes/:id is called, then the node and all its attached edges are deleted and the response includes a cascaded_edges count of removed edges
- Given a non-existent node ID, when PATCH or DELETE /api/context-graph/nodes/:id is called, then a 404 response is returned
- When a node is successfully updated via PATCH, a graph_node_updated WebSocket event is broadcast to all connected clients with the updated node payload
- When a node is successfully deleted via DELETE, a graph_node_deleted WebSocket event is broadcast to all connected clients with the node ID and cascaded_edges count
- Unit tests cover: successful update, successful delete with cascade count, 404 for unknown node on both endpoints, and WebSocket event emission for both mutations

# Architecture Spec: Context Graph — PATCH & DELETE Node Endpoints with Cascade & WebSocket Broadcasts

## Technical Approach

Add two mutation endpoints to the existing context graph router: `PATCH` and `DELETE` for nodes. Both follow the established Express router pattern (see `src/api/projects.js`). The DELETE endpoint performs a cascade-delete of all connected edges within a SQLite transaction and returns the count. Both endpoints broadcast WebSocket events via the existing `broadcast()` helper in `src/ws/broadcaster.js`. Frontend adds a node context menu, edit panel, and delete confirmation dialog — all wired to a Zustand/Redux store that also processes incoming WS events for real-time sync.

**Key design decisions:**
- Cascade delete uses an explicit `DELETE FROM context_graph_edges WHERE source_node_id = ? OR target_node_id = ?` inside a transaction (SQLite `ON DELETE CASCADE` on the FK handles this too, but we need the count *before* the node delete fires the cascade, so we delete edges first and capture `changes`)
- No new migration needed — this story assumes the `context_graph_nodes` and `context_graph_edges` tables already exist (created by the AOV-28 base migration `039_context_graph_tables.sql`)
- WebSocket events follow existing pattern: `{ type: 'graph_node_updated', ... }` / `{ type: 'graph_node_deleted', ... }`

**Open question resolutions (recommended):**
1. **Pre-fetch cascade count** — derive client-side from local edge list (no extra endpoint)
2. **Optimistic updates for PATCH** — wait for 200 response (simpler, avoids rollback complexity)
3. **Conflict handling** — if a WS `graph_node_deleted` arrives while edit panel is open for that node, show "deleted by another user" banner and close panel
4. **Node type enum** — server-defined; fetch valid types from a config endpoint or hardcode initial set
5. **Properties field** — free-form JSON; validate parse-ability only

---

## Data Model Changes

No new tables or migrations. This story operates on tables created by the prior AOV-28 migration (`039_context_graph_tables.sql`):

| Table | Relevant Columns | Notes |
|---|---|---|
| `context_graph_nodes` | `id`, `project_id`, `label`, `type`, `x`, `y`, `pinned`, `properties`, `updated_at` | PATCH updates any subset; DELETE removes row |
| `context_graph_edges` | `id`, `source_node_id`, `target_node_id` | CASCADE: edges deleted when their source or target node is deleted |

**Indexes already expected:**
- `idx_context_graph_edges_source` on `source_node_id`
- `idx_context_graph_edges_target` on `target_node_id`

These indexes are critical for performant cascade-delete lookups.

## API Changes

### PATCH /api/context-graph/nodes/:id

Update a node's mutable fields.

**Request:**
```
PATCH /api/context-graph/nodes/:id
Content-Type: application/json

{
  "label": "Updated Label",     // optional
  "type": "concept",            // optional
  "properties": { "k": "v" },  // optional, JSON object
  "x": 150,                     // optional, integer
  "y": 300,                     // optional, integer
  "pinned": true                // optional, boolean
}
```

**Response (200):**
```json
{
  "id": "uuid",
  "project_id": "uuid",
  "label": "Updated Label",
  "type": "concept",
  "x": 150,
  "y": 300,
  "pinned": 1,
  "properties": "{\"k\":\"v\"}",
  "created_at": "ISO8601",
  "updated_at": "ISO8601"
}
```

**Error responses:**
| Status | Condition | Body |
|---|---|---|
| 400 | No valid fields provided | `{ "error": "No valid fields to update" }` |
| 400 | `properties` is not valid JSON | `{ "error": "properties must be a valid JSON object" }` |
| 404 | Node not found | `{ "error": "Node not found" }` |

**WebSocket broadcast on success:**
```json
{ "type": "graph_node_updated", "projectId": "uuid", "node": { /* full updated node */ } }
```
