# AOV-45: Context graph: edge create, update, delete with validation and WebSocket

As a developer working with the context graph API, I need POST, PATCH, and DELETE endpoints for graph edges so that I can programmatically manage relationships between context nodes. The POST endpoint should validate that source and target nodes both exist, that source differs from target, and that no duplicate edge exists before persisting. PATCH allows updating the label, type, and properties of an existing edge. DELETE removes the edge. All three mutations must broadcast corresponding WebSocket events so connected clients stay in sync without polling.


## Acceptance Criteria
- Given valid source_node_id, target_node_id, label, type, and properties, when POST /api/context-graph/:projectId/edges is called, then the edge is created and returned with a 201 status
- Given source_node_id equals target_node_id, when POST /api/context-graph/:projectId/edges is called, then a 422 response is returned
- Given a node ID that does not exist in the project, when POST /api/context-graph/:projectId/edges is called, then a 422 response is returned
- Given an edge already exists between the same source and target nodes, when POST /api/context-graph/:projectId/edges is called, then a 422 response is returned
- Given an existing edge ID, when PATCH /api/context-graph/edges/:id is called with updated label, type, or properties, then the edge is updated and the changes are returned
- Given an existing edge ID, when DELETE /api/context-graph/edges/:id is called, then the edge is removed and a success response is returned; if the ID does not exist a 404 is returned
- Given any successful create, update, or delete mutation, then a graph_edge_created, graph_edge_updated, or graph_edge_deleted WebSocket event is broadcast to all subscribers of the project

## Technical Approach

Add a `graph_edges` table (migration 040) and three REST endpoints on the existing context-graph router for edge create, update, and delete. Each mutation validates business rules (no self-loops, no orphaned nodes, no duplicates) and broadcasts a typed WebSocket event via the existing `broadcast()` function. The frontend adds an `EdgeCreateModal`, `EdgeInspectorPanel`, and `EdgeDeleteConfirmDialog` that consume these endpoints and react to incoming WS events for real-time sync.

**Key constraints:**
- SQLite (better-sqlite3) — not PostgreSQL. Use `JSON` text columns for properties.
- WebSocket is global broadcast (no rooms). Clients filter by `project_id` in payload.
- ES modules throughout. Follow existing router/test patterns exactly.

## Data Model Changes

### New Table: `graph_edges` (migration `040_graph_edges.sql`)

| Column | Type | Constraints |
|---|---|---|
| `id` | TEXT | PRIMARY KEY |
| `project_id` | TEXT | NOT NULL, FK → `projects(id)` |
| `source_node_id` | TEXT | NOT NULL, FK → `graph_nodes(id)` |
| `target_node_id` | TEXT | NOT NULL, FK → `graph_nodes(id)` |
| `label` | TEXT | NOT NULL |
| `type` | TEXT | NOT NULL |
| `properties` | TEXT | DEFAULT '{}' (JSON) |
| `created_at` | TEXT | NOT NULL |
| `updated_at` | TEXT | NOT NULL |

**Indexes:**
- `idx_graph_edges_project` ON `graph_edges(project_id)`
- `idx_graph_edges_source` ON `graph_edges(source_node_id)`
- `idx_graph_edges_target` ON `graph_edges(target_node_id)`
- `UNIQUE(source_node_id, target_node_id)` — prevents duplicate edges

**CHECK constraint:**
- `CHECK(source_node_id != target_node_id)` — prevents self-loops at DB level

**Migration SQL:**
```sql
CREATE TABLE IF NOT EXISTS graph_edges (
  id              TEXT PRIMARY KEY,
  project_id      TEXT NOT NULL REFERENCES projects(id),
  source_node_id  TEXT NOT NULL REFERENCES graph_nodes(id) ON DELETE CASCADE,
  target_node_id  TEXT NOT NULL REFERENCES graph_nodes(id) ON DELETE CASCADE,
  label           TEXT NOT NULL,
  type            TEXT NOT NULL,
  properties      TEXT DEFAULT '{}',
  created_at      TEXT NOT NULL,
  updated_at      TEXT NOT NULL,
  CHECK(source_node_id != target_node_id),
  UNIQUE(source_node_id, target_node_id)
);

CREATE INDEX idx_graph_edges_project ON graph_edges(project_id);
CREATE INDEX idx_graph_edges_source  ON graph_edges(source_node_id);
CREATE INDEX idx_graph_edges_target  ON graph_edges(target_node_id);
```

> **Note:** `ON DELETE CASCADE` on both FK columns ensures edges are automatically removed when either endpoint node is deleted (supports the node delete story's cascade requirement).

## API Changes

All endpoints follow existing patterns: `getDb()` for database access, `broadcast()` for WS events, `uuidv4()` for IDs.

### POST `/api/context-graph/:projectId/edges`

Create a new edge with validation.

**Request body:**
```json
{
  "source_node_id": "uuid",
  "target_node_id": "uuid",
  "label": "depends_on",
  "type": "dependency",
  "properties": { "weight": 1 }
}
```

**Validation logic (in order):**
1. `source_node_id`, `target_node_id`, `label`, `type` are required → 400
2. `source_node_id === target_node_id` → 422 `"Source and target must be different nodes"`
3. Source node must exist in project → 422 `"Source node not found in this project"`
4. Target node must exist in project → 422 `"Target node not found in this project"`
5. No existing edge with same `(source_node_id, target_node_id)` → 422 `"An edge already exists between these two nodes"`

**Success:** 201 + created edge object
**WebSocket:** `broadcast({ type: 'graph_edge_created', edge, project_id })`
