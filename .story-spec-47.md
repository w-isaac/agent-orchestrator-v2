# AOV-47: Auto-merge compatible conflicts and re-queue incompatible tasks with structured logging

As the orchestrator system, I need to automatically resolve classified conflicts after the detection layer identifies them, so that non-overlapping and compatible changes are merged without user intervention, incompatible changes cause the offending task to be re-queued, and every resolution action is persisted for audit and debugging purposes. This reduces manual conflict triage and keeps the pipeline flowing for resolvable cases while safely deferring truly incompatible changes.


## Acceptance Criteria
- Given two tasks with non-overlapping artifact changes, when conflict resolution runs, then the changes are auto-merged and a log entry is written with resolution_action='auto_merged_non_overlapping'
- Given two tasks with compatible artifact changes, when conflict resolution runs, then the changes are auto-merged and a log entry is written with resolution_action='auto_merged_compatible'
- Given two tasks with incompatible artifact changes, when conflict resolution runs, then the completing task is re-queued with status 'conflict_requeued' and a log entry is written with resolution_action='requeued_incompatible'
- Each structured log entry contains task_id, artifact_id, classification, and resolution_action fields
- GET /api/conflict-log returns all conflict resolution events and supports an optional task_id query parameter to filter results
- Unit tests cover auto-merge for non-overlapping conflicts, auto-merge for compatible conflicts, re-queue for incompatible conflicts, and presence of correct log entries for each case

# Architecture Spec: Auto-Merge Compatible Conflicts & Re-Queue Incompatible Tasks

**Story:** Auto-merge compatible conflicts and re-queue incompatible tasks with structured logging
**Date:** 2026-04-17

---

## Technical Approach

Introduce a **conflict resolution service** that runs after the existing conflict detection/classification layer. The service receives two tasks and their classified conflict type, then:

1. **Non-overlapping / Compatible** — auto-merges artifact changes and persists a structured log entry.
2. **Incompatible** — re-queues the completing task with `status = 'conflict_requeued'` and persists a log entry.

All resolution actions are written to a new `conflict_resolution_log` table for audit. A new REST endpoint (`GET /api/conflict-log`) exposes these records. The frontend adds a Conflict Log Viewer page, dashboard badges, and a per-task Conflict History tab.

**Key design decisions:**
- Dedicated `conflict_resolution_log` table (not `system_events`) — purpose-built schema with indexed columns for the required query patterns (by `task_id`, `classification`, `resolution_action`).
- `conflict_requeued` is treated as a distinct story status alongside existing statuses (`queued`, `in_progress`, `gate`, `done`, `failed`). The scheduler already skips non-`queued` stories, so `conflict_requeued` tasks will only re-enter the pipeline when explicitly transitioned back to `queued` by the resolution service after conflict conditions clear.
- The resolution service is invoked from the scheduler tick when artifact overlap is detected between concurrent tasks — it slots in after classification and before the task's agent commit is finalized.

## Data Model Changes

### New Table: `conflict_resolution_log`

**Migration:** `039_conflict_resolution_log.sql`

```sql
CREATE TABLE IF NOT EXISTS conflict_resolution_log (
  id                  TEXT PRIMARY KEY,
  task_id             TEXT NOT NULL REFERENCES stories(id),
  artifact_id         TEXT NOT NULL,
  classification      TEXT NOT NULL,       -- 'non_overlapping' | 'compatible' | 'incompatible'
  resolution_action   TEXT NOT NULL,       -- 'auto_merged_non_overlapping' | 'auto_merged_compatible' | 'requeued_incompatible'
  conflicting_task_id TEXT REFERENCES stories(id),
  created_at          TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_conflict_log_task_id    ON conflict_resolution_log(task_id);
CREATE INDEX IF NOT EXISTS idx_conflict_log_created_at ON conflict_resolution_log(created_at);
CREATE INDEX IF NOT EXISTS idx_conflict_log_action     ON conflict_resolution_log(resolution_action);
```

| Column | Type | Description |
|---|---|---|
| `id` | TEXT PK | UUID |
| `task_id` | TEXT FK → stories | The task whose conflict was resolved |
| `artifact_id` | TEXT | Artifact path/ID involved in the conflict |
| `classification` | TEXT | `non_overlapping`, `compatible`, `incompatible` |
| `resolution_action` | TEXT | `auto_merged_non_overlapping`, `auto_merged_compatible`, `requeued_incompatible` |
| `conflicting_task_id` | TEXT FK → stories (nullable) | The peer task that caused the conflict |
| `created_at` | TEXT | ISO 8601 timestamp |

### Modified: `stories` table

No schema migration needed — `status` is an unconstrained `TEXT` column. The new value `'conflict_requeued'` is introduced at the application level. The scheduler's `tick()` and the PATCH endpoint already handle arbitrary status strings.

**New status value:** `conflict_requeued` — set when an incompatible conflict causes the task to be re-queued. The resolution service transitions it back to `queued` when the conflicting task completes or the conflict is manually resolved.
