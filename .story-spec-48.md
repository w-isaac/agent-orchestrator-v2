# AOV-48: ForceGraph context menus and node detail panel

As a user interacting with the ForceGraph canvas, I want right-click context menus on the canvas, nodes, and edges so that I can quickly perform actions (add, edit, delete) without leaving the graph view. I also want a sidebar detail panel that appears when I select a node, showing all its properties alongside Edit and Delete actions, so I can inspect and manage node data in context. This extends the existing ForceGraph component built in AOV-28.


## Acceptance Criteria
- Given the user right-clicks on empty canvas space, a context menu appears with an Add Node option
- Given the user right-clicks on a node, a context menu appears with Edit Node, Delete Node, and Add Edge From Here options
- Given a node is selected, the NodeDetailPanel sidebar displays all node properties with Edit and Delete action buttons
- Given the user initiates a delete action on a node or edge, an InlineConfirmTooltip is shown and the deletion only proceeds after explicit confirmation

# Architecture Spec: ForceGraph Context Menus & Node Detail Panel

## Technical Approach

Add graph data persistence (nodes + edges) to the orchestrator backend with full CRUD APIs and WebSocket broadcast for real-time sync. The frontend (separate client repo) will consume these APIs from the existing ForceGraph canvas, adding context menus (canvas, node, edge), a node detail sidebar panel, inline delete confirmation, and edge-draw mode.

**Key decisions:**
- Graph data is **project-scoped** — each project has its own node/edge graph
- Node properties stored as JSON blob for schema flexibility
- SQLite + better-sqlite3 (matching existing stack) — no new DB engine
- WebSocket events broadcast on every graph mutation for multi-client sync
- Backend is the sole scope of this repo; frontend components are documented for the client repo

#### `graph_edges` (Migration 039, same file)

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | TEXT | PRIMARY KEY | UUID v4 |
| `project_id` | TEXT | NOT NULL, FK → projects(id) ON DELETE CASCADE | |

| `type` | TEXT | DEFAULT NULL | Optional edge type |
| `properties` | TEXT | DEFAULT '{}' | JSON key-value bag |
| `created_at` | TEXT | NOT NULL | ISO 8601 |
| `updated_at` | TEXT | NOT NULL | ISO 8601 |

**Indexes:**
- `idx_graph_edges_project` ON `graph_edges(project_id)`
- `idx_graph_edges_source` ON `graph_edges(source_node_id)`
- `idx_graph_edges_target` ON `graph_edges(target_node_id)`

**Constraint:** `CHECK (source_node_id != target_node_id)` — no self-loops.

## API Changes

All endpoints scoped under `/api/projects/:projectId/graph`.

### Graph — Full Load

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/projects/:projectId/graph` | Returns `{ nodes: [...], edges: [...] }` for the project |

**Response:**
```json
{
  "nodes": [{ "id", "label", "type", "x", "y", "properties": {}, "created_at", "updated_at" }],
  "edges": [{ "id", "source_node_id", "target_node_id", "label", "type", "properties": {}, "created_at", "updated_at" }]
}
```

### Nodes CRUD

| Method | Path | Request Body | Response | WS Event |
|--------|------|-------------|----------|----------|
| `POST` | `.../graph/nodes` | `{ label, type?, x?, y?, properties? }` | `201` + node | `graph_node_created` |
| `PATCH` | `.../graph/nodes/:nodeId` | `{ label?, type?, x?, y?, properties? }` | `200` + node | `graph_node_updated` |
| `DELETE` | `.../graph/nodes/:nodeId` | — | `204` | `graph_node_deleted` |

**DELETE cascade:** Deleting a node also deletes all connected edges (DB cascade). The server response includes `deletedEdgeIds: [...]` so clients can remove them without a refetch. WS event includes same.
