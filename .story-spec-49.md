# AOV-49: ForceGraph node/edge modals and drag-to-edge creation

As a user working with the context graph, I need interactive modals to create and edit nodes and edges, as well as an intuitive drag-from-handle gesture for creating edges between nodes. This allows me to build and refine the context graph directly in the UI without manual API calls. The NodeEditModal lets me set a label, type, and arbitrary key-value properties. The EdgeEditModal lets me set a label and type for a relationship. When I drag from a node's edge-handle to another node, a dashed-line preview follows my cursor, and releasing creates the edge and opens the EdgeEditModal pre-filled with the source and target nodes. This builds on the context menus introduced in AOV-48.


## Acceptance Criteria
- Given the NodeEditModal is open, the user can set a label, type, and arbitrary key-value properties, and save to persist the node via API
- Given the EdgeEditModal is open, the user can set a label and type for the edge, and save to persist the edge via API
- Given the user drags from a node edge-handle to another node, a dashed-line preview follows the cursor and releasing creates an edge, opening the EdgeEditModal pre-filled with source and target

## Screen Descriptions

### 1. ForceGraph Canvas (modified)

**Layout:** Full-viewport SVG/canvas. Nodes rendered as circles with label beneath. Edges rendered as lines with optional label.

**New element — Edge Handle:**
- A small circular affordance (8px radius, semi-transparent) appears on node hover, positioned at the node's boundary (e.g., right edge at 3 o'clock).
- Cursor changes to `crosshair` on handle hover.
- Handle color: accent (e.g., `#4F9DFF`), with a subtle pulse animation on first hover to aid discoverability.

**States:**
| State | Description |
|---|---|
| Idle | Nodes and edges visible; no handles shown |
| Node hovered | Edge handle appears on the hovered node |
| Dragging (in-progress) | Dashed-line preview from source node to cursor; source node highlighted with ring; target nodes highlight on cursor proximity |
| Dragging over valid target | Target node gets a green highlight ring |
| Dragging over invalid target (self) | Target node gets red highlight; drop is rejected |
| Edge dropped | EdgeEditModal opens; preview line removed |

## Technical Approach

Extend the existing Agent Orchestrator with:

1. **Two new database tables** (`graph_nodes`, `graph_edges`) storing project-scoped context graph data, with a JSON `properties` column on nodes.
2. **CRUD API endpoints** under `/api/projects/:projectId/graph/` for nodes and edges, plus type-listing endpoints.
3. **Four new React components** (`NodeEditModal`, `EdgeEditModal`, `EdgeDragPreview`, `PropertyRow`) integrated into the existing `ForceGraph` canvas, leveraging modal state and drag state managed at the ForceGraph level.
4. **Drag-to-edge interaction** using SVG mouse/touch events with a dashed-line preview rendered in SVG coordinate space.

The modals are triggered from the context menus built in AOV-48. Node creation/editing persists via `POST`/`PATCH` on nodes; edge creation/editing via `POST`/`PATCH` on edges. Drag-drop creates the edge then opens `EdgeEditModal` pre-filled.

## Data Model Changes

### New Migration: `src/db/migrations/037_graph_nodes_edges.sql`

```sql
-- Context graph: nodes
CREATE TABLE IF NOT EXISTS graph_nodes (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  label       TEXT NOT NULL,
  type        TEXT,
  properties  TEXT DEFAULT '{}',   -- JSON object of arbitrary key-value pairs
  x           REAL,                -- persisted position (optional)
  y           REAL,
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_graph_nodes_project ON graph_nodes(project_id);
CREATE INDEX IF NOT EXISTS idx_graph_nodes_type    ON graph_nodes(type);

-- Context graph: edges
CREATE TABLE IF NOT EXISTS graph_edges (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  source_id   TEXT NOT NULL REFERENCES graph_nodes(id) ON DELETE CASCADE,
  target_id   TEXT NOT NULL REFERENCES graph_nodes(id) ON DELETE CASCADE,
  label       TEXT,
  type        TEXT,
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_graph_edges_project ON graph_edges(project_id);
CREATE INDEX IF NOT EXISTS idx_graph_edges_source  ON graph_edges(source_id);
CREATE INDEX IF NOT EXISTS idx_graph_edges_target  ON graph_edges(target_id);
```

**Key decisions:**
- `properties` stored as JSON text (better-sqlite3 has no native JSON type; parsed in application layer).
- `x`/`y` are optional — allows persisting user-positioned nodes while still supporting force layout.
- `ON DELETE CASCADE` on `project_id` and node foreign keys ensures cleanup.
- No unique constraint on `(source_id, target_id)` — parallel edges are allowed (see Open Question #5 in UX spec).
