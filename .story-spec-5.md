# AOV-5: Implement three-phase context retrieval: graph traversal, embedding re-ranking, and budget packing

As an agent dispatching tasks, I need a three-phase context retrieval system so that I receive the most relevant and token-efficient context payload for each task. The system must: (1) traverse the context graph from seed nodes using configurable edge-type scores with depth decay; (2) re-rank candidates using a weighted combination of graph proximity (0.6) and embedding cosine similarity (0.4); and (3) greedily pack the token budget using a knapsack algorithm with multi-resolution fallback (full → summary → skip). The result is a tiered context payload that maximizes relevance within the configured token budget.


## Acceptance Criteria
- Given a set of seed node IDs and a configured token budget, when context retrieval is triggered, then the system performs graph traversal from each seed node up to a configurable max depth, scoring edges by type with a depth decay factor applied at each hop
- Given traversal results, when re-ranking is applied, then each candidate node receives a combined score of (0.6 × graph_proximity_score) + (0.4 × embedding_cosine_similarity) against the task query embedding
- Given re-ranked candidates and a token budget, when budget packing runs, then nodes are greedily selected using a knapsack approach: first attempting full content, falling back to summary, then skipping if neither fits
- Given a configurable token budget per task, when the budget is exceeded mid-packing, then the system applies multi-resolution fallback without dropping higher-ranked nodes unnecessarily
- Given the retrieval pipeline completes, when results are returned, then the payload is a tiered structure distinguishing full-content nodes from summary nodes
- Given any retrieval call, when edge-type scores and depth decay factor are not specified, then sensible defaults are used and the configuration is documented
- Given the implementation, when unit tests run against a sample graph with known embeddings, then all three phases are independently testable and produce deterministic results

## Screen Descriptions

### 1. Retrieval Configuration Panel

**Route:** `/settings/context-retrieval` or embedded in task dispatch settings

**Layout:** Single-column form, sectioned by phase. Sticky "Save & Apply" footer.

**Sections:**

**Phase 1 — Graph Traversal**
- `max_depth` — number input (default: 3, range 1–10)
- `depth_decay_factor` — decimal slider (0.0–1.0, default: 0.8, step 0.05), displays live formula: `score × decay^depth`
- Edge Type Scoring table:
  - Columns: Edge Type | Weight | Enabled toggle
  - Rows: one per registered edge type (e.g. `DEPENDS_ON`, `REFERENCES`, `IMPLEMENTS`)
  - Inline add-row button at bottom
  - Row delete (×) on hover

**Phase 2 — Re-ranking Weights**
- Two locked inputs showing weight distribution: Graph Proximity `0.6` / Embedding Similarity `0.4`
- Info tooltip: "Weights sum to 1.0. Customization available in future release."

**Phase 3 — Budget Packing**
- `token_budget` — number input with unit label "tokens" (default: 4096)
- Fallback order display: read-only ordered list `Full → Summary → Skip`
- `prefer_higher_ranked` — checkbox, default checked

**States:**
- **Default/populated:** shows current saved config
- **Unsaved changes:** "Save & Apply" button activates; dirty indicator on section headers
- **Saving:** spinner in button, inputs disabled
- **Error:** inline field-level validation messages (e.g. decay must be 0–1)
- **First-run (no config saved):** pre-filled defaults with banner: "These are the recommended defaults — modify as needed."

- **5 test files** covering unit tests per phase + integration tests for the full pipeline and API layer
