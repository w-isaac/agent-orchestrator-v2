# AOV-50: ForceGraph optimistic mutations with error toast rollback

As a user interacting with the context graph, I want all create, edit, and delete actions to apply immediately to the graph (optimistic UI) so that the interface feels responsive. If the underlying API call fails, the change should be rolled back and I should see a visible ErrorToast notification. When multiple failures occur in quick succession, toasts should stack and each auto-dismiss after 6 seconds, so I am always informed of failures without the UI becoming permanently broken.


## Acceptance Criteria
- Given a user creates a node or edge, when the action is triggered, then the graph is updated immediately before the API response is received
- Given a user edits a node or edge, when the action is triggered, then the updated state is reflected in the graph immediately before the API response is received
- Given a user deletes a node or edge, when the action is triggered, then the element is removed from the graph immediately before the API response is received
- Given any mutating action (create/edit/delete) results in an API failure, when the error is received, then the graph is rolled back to its pre-mutation state and an ErrorToast is displayed
- Given multiple API failures occur in quick succession, when errors are received, then ErrorToasts stack visually and each auto-dismisses after 6 seconds

# Architecture Spec: ForceGraph Optimistic Mutations with Error Toast Rollback

## Technical Approach

Implement optimistic UI for all graph mutations (create/edit/delete of nodes and edges) in the React client, backed by a snapshot-based rollback mechanism. The existing Express REST API and WebSocket broadcast infrastructure remain unchanged — all new work is frontend-only.

**Strategy:**
- Introduce a `useOptimisticGraph` hook that wraps every mutation call with: (1) snapshot current graph state, (2) apply optimistic change, (3) fire API call, (4) on failure → restore snapshot + emit toast
- New `ErrorToast` and `ToastStack` components render in a fixed overlay region
- Pending mutations are tracked by ID so the `ForceGraphCanvas` can apply dashed-border/opacity treatment
- No backend changes required — existing endpoints already support all CRUD operations

**Key Architectural Decision:** Per-mutation snapshots (not global undo stack). Each in-flight mutation captures only the slice of state it touches. This avoids conflicts when multiple mutations are in flight simultaneously.

## Data Model Changes

**None.** The existing schema fully supports all required operations:

| Table | Role | Existing Columns Used |
|---|---|---|
| `stories` | Graph nodes | `id`, `title`, `description`, `status`, `current_stage_id`, `project_id` |
| `story_dependencies` | Graph edges | `story_id`, `depends_on` (composite PK) |

No new tables, columns, indexes, or migrations are needed. All optimistic state is managed client-side in React state.

## API Changes

**No new or modified endpoints.** The existing API surface is sufficient:

| Operation | Method | Path | Request Body | Response |
|---|---|---|---|---|
| Create node | `POST` | `/api/stories` | `{ project_id, title, description, ... }` | `201 { id, ... }` |
| Edit node | `PATCH` | `/api/stories/:id` | `{ title?, description?, ... }` | `200 { id, ... }` |
| Delete node | `DELETE` | `/api/stories/:id` | — | `204` |
| Set edges | `PUT` | `/api/stories/:id/dependencies` | `{ depends_on: [id, ...] }` | `200 { depends_on }` |
| Get graph | `GET` | `/api/projects/:id/dependencies` | — | `200 [{ story_id, depends_on }]` |
| Get nodes | `GET` | `/api/projects/:id/pipeline` | — | `200 { project, pipeline }` |

**Error contract note:** API errors currently return `{ error: string }`. The toast message should display this string directly. No backend changes to error format are required, but see Open Question #2 in the UX spec regarding generic vs. specific messages.
