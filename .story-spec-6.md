# AOV-6: JSON import/export for context graph

As a developer or system integrator, I need to import and export the context graph as JSON so that I can migrate data between environments, back up project graphs, and seed new projects from existing structured knowledge. On import, the system reads nodes and edges from a JSON interchange file, validates all node references, persists them to context_nodes and context_edges tables, and auto-generates embeddings and summaries for each node. On export, the system serialises the full project graph (nodes + edges) into the JSON interchange format for downstream consumption.


## Acceptance Criteria
- Given a valid JSON file with nodes and edges, when a POST /api/graphs/:projectId/import is called, then all nodes are created in context_nodes and all edges are created in context_edges with correct references
- Given a JSON import payload, when node references in edges do not match any node in the payload or the database, then the import is rejected with a descriptive validation error before any records are written
- Given a successful import, when nodes are persisted, then embeddings and summaries are auto-generated for each imported node (matching the behaviour of the standard ingestion pipeline)
- Given a project with an existing context graph, when GET /api/graphs/:projectId/export is called, then the response contains a JSON document with all nodes and edges in the interchange format
- Given the import/export operations, when they are triggered via CLI command (e.g. `graph import <file>` / `graph export <projectId>`), then the operations complete successfully and report node/edge counts
- Given an empty or malformed JSON import file, when import is attempted, then the API returns a 400 error with a clear message and no partial data is written

## ARCHITECTURE

Architecture spec written to `/app/.arch-spec-0a94f221-output.md`.

Here's a summary of the key architectural decisions:

**Data Model:** 2 new tables — `context_nodes` (with ref_id, type, label, content, embedding, summary) and `context_edges` (source/target FKs, relationship, weight). Migration `039_context_graph.sql`.

**API:** 3 endpoints on a new `/api/graphs` router:
- `POST /:projectId/import` — transactional import with pre-write validation
- `GET /:projectId/export` — full graph serialization
- `GET /:projectId/counts` — lightweight count query for the export dialog

**Key design choices:**
- **No new dependencies** — JSON body via `express.json()`, file reads via native `fs`, client file handling via `FileReader` API
- **Transactional imports** — entire import in one SQLite transaction; validation failure = full rollback, zero partial writes
- **Embedding generation post-commit** — nodes persist first, embeddings generated after via `claude-runner.js`, with graceful failure (flagged for retry)
- **Reject on conflict** as default merge strategy until product decides on merge semantics

**5 risks identified:** large graph perf, concurrent import conflicts, merge/replace ambiguity, embedding failures, oversized payloads — all with documented mitigations.
