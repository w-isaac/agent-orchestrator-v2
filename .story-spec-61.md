# AOV-61: Gatekeeper: reversible DB migrations for reviews, dependencies, and story alterations

As a backend engineer, I need three sequenced, fully reversible database migrations to support the Gatekeeper feature: (1) a gatekeeper_reviews table for storing evaluation results with deduplication via evaluation_hash, (2) a story_dependencies table for tracking inter-story dependency edges with self-reference protection, and (3) alterations to the stories table adding parent_story_id, gatekeeper_status, is_split_replaced, and split_at columns plus extension of the status enum to include pre_engineering, revision_required, and split_replaced values. The enum extension must use the rename-swap pattern to remain safely reversible. Rollback of migration 003 must abort with a clear error if any row uses a new enum value.


## Acceptance Criteria
- Given migration 20260418_001 is run, then a gatekeeper_reviews table exists with a unique index on (story_id, evaluation_hash) and an index on created_at DESC; pgcrypto is enabled via CREATE EXTENSION IF NOT EXISTS if not already installed
- Given migration 20260418_002 is run, then a story_dependencies table exists with a unique constraint on (story_id, depends_on_story_id) and a CHECK constraint preventing self-referencing rows
- Given migration 20260418_003 is run, then the stories table has columns parent_story_id, gatekeeper_status, is_split_replaced, and split_at, and the status enum includes pre_engineering, revision_required, and split_replaced values
- Given all three migrations are applied to a fresh database, then each migration can be rolled back independently without errors
- Given migration 20260418_003 down is executed when any story row uses a new enum value (pre_engineering, revision_required, or split_replaced), then the rollback aborts with a clear, descriptive error message

## ARCHITECTURE

# Architecture Specification — Gatekeeper Reversible DB Migrations

## Technical Approach

Three sequenced, independently reversible SQL migrations under `migrations/` using the project's existing migration runner (node-pg-migrate or Knex-style up/down pairs). All DDL wrapped in transactions where PostgreSQL permits. Enum extension uses the **rename-swap pattern**: create new enum type with expanded values, alter column to use new type via `USING` cast, drop old type, rename new to old. Rollback of 003 performs the reverse swap but first executes a guard query that aborts with `RAISE EXCEPTION` if any row holds a new enum value.

No API or frontend code is produced in this story — it is schema-only. UI work from the UX spec is covered by downstream stories that depend on this migration landing first.

## Data Model Changes

### Migration 20260418_001 — `gatekeeper_reviews`

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE gatekeeper_reviews (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  story_id        UUID NOT NULL REFERENCES stories(id) ON DELETE CASCADE,
  evaluation_hash TEXT NOT NULL,
  result          TEXT NOT NULL,           -- pass | fail | warning
  summary         TEXT,
  details         JSONB,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX ux_gk_reviews_story_hash
  ON gatekeeper_reviews (story_id, evaluation_hash);
CREATE INDEX ix_gk_reviews_created_at_desc
  ON gatekeeper_reviews (created_at DESC);
```

### Migration 20260418_002 — `story_dependencies`

```sql
CREATE TABLE story_dependencies (
  id                     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  story_id               UUID NOT NULL REFERENCES stories(id) ON DELETE CASCADE,
  depends_on_story_id    UUID NOT NULL REFERENCES stories(id) ON DELETE CASCADE,
  created_at             TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_story_dep UNIQUE (story_id, depends_on_story_id),
  CONSTRAINT chk_no_self_ref CHECK (story_id <> depends_on_story_id)
);
CREATE INDEX ix_story_dep_depends_on ON story_dependencies (depends_on_story_id);
```

### Migration 20260418_003 — `stories` alterations + enum extension

```sql
-- 1. Add columns
ALTER TABLE stories
  ADD COLUMN parent_story_id   UUID REFERENCES stories(id) ON DELETE SET NULL,
  ADD COLUMN gatekeeper_status TEXT,
  ADD COLUMN is_split_replaced BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN split_at          TIMESTAMPTZ;

CREATE INDEX ix_stories_parent_story_id ON stories (parent_story_id);
CREATE INDEX ix_stories_is_split_replaced
  ON stories (is_split_replaced) WHERE is_split_replaced = TRUE;

-- 2. Enum rename-swap
CREATE TYPE story_status_new AS ENUM (
  <existing values...>,
  'pre_engineering', 'revision_required', 'split_replaced'
);
ALTER TABLE stories
  ALTER COLUMN status TYPE story_status_new
  USING status::text::story_status_new;
DROP TYPE story_status;
ALTER TYPE story_status_new RENAME TO story_status;
```

**Down (003) — rollback guard:**

```sql
DO $$
DECLARE blocking_count INT;
BEGIN
  SELECT COUNT(*) INTO blocking_count FROM stories
    WHERE status IN ('pre_engineering','revision_required','split_replaced');
  IF blocking_count > 0 THEN
    RAISE EXCEPTION 'Cannot rollback migration 20260418_003: % story row(s) use new enum values (pre_engineering/revision_required/split_replaced). Migrate data first.', blocking_count;
  END IF;
END $$;
-- then reverse the rename-swap and drop columns
```

## Risk Assessment

| # | Risk | Mitigation |
|---|---|---|
| 1 | Enum rename-swap drops old type — any view/function referencing old type breaks | Pre-audit `pg_depend`; rebuild dependents in-migration |
| 2 | Rollback 003 data-loss if rows hold new enum values | `RAISE EXCEPTION` guard block aborts before DDL |
| 3 | Long `ALTER TYPE ... USING` rewrite on large stories table | Run in maintenance window; table rewrite is `ACCESS EXCLUSIVE` |
| 4 | `evaluation_hash` collision across unrelated stories acceptable (scoped to story_id) — confirmed by unique composite | Composite uniqueness documented |
| 5 | Self-reference in `story_dependencies` at app layer pre-CHECK | CHECK constraint enforces at DB; app validates for UX |
| 6 | Concurrent writes during migration 003 enum swap | Take `ACCESS EXCLUSIVE` briefly; deploy during quiet window |

**Backward compatibility:** Additive columns are nullable or have defaults; existing reads unaffected. Enum values are append-only — existing code paths unchanged.

## Testing Strategy

**Unit (migration runner):**
- Each up script applies cleanly on a fresh schema.
- Each down script reverses cleanly when preconditions met.
- `CHECK (story_id <> depends_on_story_id)` rejects self-refs (insert test).
- Unique index on `(story_id, evaluation_hash)` rejects duplicates.

**Integration (full sequence on ephemeral Postgres):**
1. Apply 001→002→003 on fresh DB; assert schema via `information_schema` queries.
2. Rollback 003→002→001 independently; assert clean teardown.
3. Re-apply after rollback (idempotency / repeatability).
4. Assert enum contains new values post-003; assert removed post-003-down.

**Rollback-guard e2e:**
- Seed a `stories` row with `status = 'pre_engineering'`.
- Execute 003 down; assert the migration aborts with the descriptive `RAISE EXCEPTION` message and the schema is unchanged.
- Update seeded row to a legacy status; re-run down; assert success.

**Validation queries:**
- `SELECT enum_range(NULL::story_status);` contains the 3 new values after 003.
- `pg_indexes` shows `ux_gk_reviews_story_hash` and `ix_gk_reviews_created_at_desc`.
- `pg_constraint` shows `chk_no_self_ref` and `uq_story_dep`.
