# AOV-73: DDL migration: port columns with CHECK constraints, partial unique indexes, and integration tests

As a platform engineer, I need a reversible database migration that adds four nullable INTEGER port columns (deploy_port, frontend_port, backend_port, container_port) to the projects table, each with a CHECK constraint enforcing valid port range (1–65535) and a partial unique index allowing multiple NULLs but rejecting duplicate non-NULL values. Integration tests against a real Postgres instance must verify column existence, constraint rejection, uniqueness enforcement, NULL permissiveness, and a full up→down→up round-trip so that downstream services can safely store and query per-project port configuration.


## Acceptance Criteria
- Given the up migration is applied, when querying the projects table schema, then columns deploy_port, frontend_port, backend_port, and container_port exist as nullable INTEGER columns
- Given a port column exists, when inserting a row with a port value outside 1–65535 (e.g. 0 or 65536), then Postgres rejects the insert with SQLSTATE 23514
- Given a port column exists, when two rows have the same non-NULL port value in the same column, then the second insert is rejected with SQLSTATE 23505; when multiple rows have NULL in the same column, all inserts succeed
- Given the up migration has been applied, when the down migration is run, then all four port columns and their indexes are cleanly removed; re-applying the up migration fully restores them (round-trip passes)
- Given the migration file, when the integration test suite runs against a real Postgres instance, then all tests covering column existence, CHECK rejection, unique enforcement, NULL permissiveness, and round-trip pass with no failures

- **container_port**: 3000

# Architecture Specification: Per-Project Port Configuration

## Technical Approach

Add four nullable INTEGER port columns to `projects` via a reversible DDL migration with CHECK constraints (1–65535) and partial unique indexes (WHERE col IS NOT NULL). Expose ports via a dedicated REST sub-resource on projects. Surface validation errors (SQLSTATE 23514 → range; 23505 → uniqueness) as structured API errors mapped to UI states. Integration tests run against a real Postgres (Docker) to verify existence, CHECK rejection, UNIQUE enforcement, NULL permissiveness, and up→down→up round-trip.

## Data Model Changes

**Table altered:** `projects`

| Column | Type | Nullable | Constraint |
|---|---|---|---|
| `deploy_port` | INTEGER | YES | CHECK (deploy_port BETWEEN 1 AND 65535) |
| `frontend_port` | INTEGER | YES | CHECK (frontend_port BETWEEN 1 AND 65535) |
| `backend_port` | INTEGER | YES | CHECK (backend_port BETWEEN 1 AND 65535) |
| `container_port` | INTEGER | YES | CHECK (container_port BETWEEN 1 AND 65535) |

**Partial unique indexes:**
- `ux_projects_deploy_port ON projects(deploy_port) WHERE deploy_port IS NOT NULL`
- `ux_projects_frontend_port ON projects(frontend_port) WHERE frontend_port IS NOT NULL`
- `ux_projects_backend_port ON projects(backend_port) WHERE backend_port IS NOT NULL`
- `ux_projects_container_port ON projects(container_port) WHERE container_port IS NOT NULL`

**Migration (up):**
```sql
ALTER TABLE projects
  ADD COLUMN deploy_port    INTEGER CHECK (deploy_port    BETWEEN 1 AND 65535),
  ADD COLUMN frontend_port  INTEGER CHECK (frontend_port  BETWEEN 1 AND 65535),
  ADD COLUMN backend_port   INTEGER CHECK (backend_port   BETWEEN 1 AND 65535),
  ADD COLUMN container_port INTEGER CHECK (container_port BETWEEN 1 AND 65535);

CREATE UNIQUE INDEX ux_projects_deploy_port    ON projects(deploy_port)    WHERE deploy_port    IS NOT NULL;
CREATE UNIQUE INDEX ux_projects_frontend_port  ON projects(frontend_port)  WHERE frontend_port  IS NOT NULL;
CREATE UNIQUE INDEX ux_projects_backend_port   ON projects(backend_port)   WHERE backend_port   IS NOT NULL;
CREATE UNIQUE INDEX ux_projects_container_port ON projects(container_port) WHERE container_port IS NOT NULL;
```

**Migration (down):**
```sql
DROP INDEX IF EXISTS ux_projects_deploy_port, ux_projects_frontend_port,
                     ux_projects_backend_port, ux_projects_container_port;
ALTER TABLE projects
  DROP COLUMN IF EXISTS deploy_port,
  DROP COLUMN IF EXISTS frontend_port,
  DROP COLUMN IF EXISTS backend_port,
  DROP COLUMN IF EXISTS container_port;
```

## Risk Assessment

1. **Global uniqueness semantics** — spec open question #1: migration enforces global uniqueness across all projects. Confirm before release; scoping by environment would require composite partial index.
2. **Concurrent writes** — two simultaneous PUTs claiming the same port: PG will reject one with 23505; ensure handler returns 409 atomically (no retry loop).
3. **Index creation on large `projects` table** — `CREATE UNIQUE INDEX` locks writes. For prod use `CREATE UNIQUE INDEX CONCURRENTLY` in a separate migration step (cannot run in a transaction).
4. **Down-migration in prod** — columns disappear; UI must degrade (hide Ports tab if `GET /ports` returns 404/column-missing).
5. **Constraint-name parsing fragility** — server relies on index names to identify which field conflicts. Keep index names stable; add unit test for error translator.

## Testing Strategy

**Unit (server):**
- `portsService` PG error translator: 23514 → `PORT_OUT_OF_RANGE`, 23505 with each index name → correct field.
- Request validation: rejects non-integer, rejects >65535 client-side.

**Unit (client):**
- `PortInput` client validation (range, non-integer, null allowed).
- `PortFieldGroup` dirty tracking, reset, save dispatch.
- `PortConflictBanner` renders field name, dismiss fires callback; `PortRangeError` inline display.

**Integration (real Postgres, per AC):**
- Column existence: `information_schema.columns` shows 4 columns, type INTEGER, is_nullable YES.
- CHECK rejection: INSERT with 0 and 65536 → SQLSTATE 23514 (all four columns).
- UNIQUE enforcement: two rows with same non-NULL value → second fails 23505.
- NULL permissiveness: N rows with NULL in same column all succeed.
- Round-trip: up → assert schema → down → assert columns/indexes gone → up → assert schema restored; data inserted after second up obeys constraints.

**E2E (optional):**
- Ports tab: enter valid value → save → toast; enter 70000 → inline error; enter duplicate → `PortConflictBanner`; clear field → saves NULL.

**Validation:** All five acceptance criteria map 1:1 to integration test cases above.
