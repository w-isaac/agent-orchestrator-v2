# AOV-79: pgErrorMapper + Zod port schemas (server lib)

As a backend engineer, I need a centralized server-side library that (1) translates PostgreSQL error codes 23514 and 23505 on named port constraints into structured error response objects, and (2) exports shared Zod schemas for port values and project creation/mutation payloads — so that all route handlers can rely on a single, tested source of truth for validation and DB error formatting without duplicating logic.


## Acceptance Criteria
- portSchema accepts integers 1–65535 and null, and rejects 0, 65536, non-integers (e.g. 3.5), strings, and negative numbers
- createProjectSchema and patchProjectSchema expose port fields as optional nullable integers using portSchema
- pgErrorMapper maps PostgreSQL error code 23514 on constraints matching projects_<col>_port_range_check to HTTP 400 with body { error: 'invalid_port_range', field, message }
- pgErrorMapper maps PostgreSQL error code 23505 on constraints matching projects_<col>_port_unique to HTTP 409 with body { error: 'port_conflict', field, value }
- pgErrorMapper rethrows any unrecognised constraint/error code unchanged
- Unit tests cover the full mapper constraint-name table (all mapped names + unknown-constraint rethrow) and all portSchema edge cases (valid and invalid inputs)


## Project Infrastructure Settings

- **deploy_port**: 5001 (use this as the host port in docker-compose.yml: `5001:3000`)
- **container_port**: 3000

# Architecture Specification — pgErrorMapper + Zod Port Schemas

## Technical Approach

Create a small, pure server-side library under `server/lib/` exposing:

1. **`portSchema`** — a Zod schema for port values (integer 1–65535 or null).
2. **`createProjectSchema` / `patchProjectSchema`** — Zod schemas reusing `portSchema` for project payload validation.
3. **`pgErrorMapper`** — a function that inspects a caught `pg` error, matches PostgreSQL codes `23514`/`23505` against constraint-name regex patterns, and returns a structured `{ status, body }` object, or rethrows.

Route handlers wrap DB calls in `try/catch`, pass caught errors through `pgErrorMapper`, and respond using its output. Pure functions, no I/O, fully unit-testable.

## Data Model Changes

**None.** This story is library-only. It depends on (but does not create) existing constraints:

| Constraint Pattern | Type | PG Code |
|---|---|---|
| `projects_<col>_port_range_check` | CHECK | 23514 |
| `projects_<col>_port_unique` | UNIQUE | 23505 |

Verify these already exist on the `projects` table; if missing, a separate migration story is required.

## API Changes

**No new endpoints.** Existing project endpoints gain normalized error bodies:

| Endpoint | Scenario | Status | Body |
|---|---|---|---|
| `POST /api/projects` | Zod fails | 400 | Zod error (existing behavior) |
| `POST /api/projects` | `23514` range | 400 | `{ error: 'invalid_port_range', field, message }` |
| `POST /api/projects` | `23505` unique | 409 | `{ error: 'port_conflict', field, value }` |
| `PATCH /api/projects/:id` | same mapping | — | same |
| any | unrecognized | rethrown | unchanged |

`field` is the `<col>` captured from the constraint name. `value` is the port attempted (from request payload).

## File Changes

**Create**

| File | Purpose |
|---|---|
| `server/lib/portSchema.ts` | Export `portSchema = z.union([z.number().int().min(1).max(65535), z.null()])`. Reject non-integers explicitly. |
| `server/lib/projectSchemas.ts` | Export `createProjectSchema`, `patchProjectSchema`; port-typed fields use `portSchema.optional()`. |
| `server/lib/pgErrorMapper.ts` | Export `mapPgError(err, payload?)`. Regex table for `projects_(\w+)_port_range_check` / `projects_(\w+)_port_unique`. Rethrow on miss. |
| `server/lib/__tests__/portSchema.test.ts` | Edge cases. |
| `server/lib/__tests__/projectSchemas.test.ts` | Composition. |
| `server/lib/__tests__/pgErrorMapper.test.ts` | Full constraint-name table + rethrow. |

**Modify**

| File | Change |
|---|---|
| `server/routes/projects.ts` (or equivalent) | Import schemas; validate body; wrap DB calls with `try/catch` → `mapPgError`. |
| `server/lib/index.ts` (if barrel exists) | Re-export new modules. |

## Dependencies

**None new.** Assumes `zod` and `pg` already present. If `zod` is absent: add `zod` (^3.x) — de facto standard for TS schema validation, tree-shakeable, no runtime deps.

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Constraint names drift from regex (e.g. renamed migration) | Anchor regex `^projects_([a-z_]+)_port_(range_check\|unique)$`; cover with snapshot test of actual DB constraint names in integration. |
| Rethrow loses stack | Rethrow the original `err` object untouched, never wrap. |
| Zod `z.number()` accepts `NaN`/floats | Use `.int()` + `.finite()`; explicitly test `3.5`, `NaN`, `"80"`. |
| `value` leaking PII in `port_conflict` | Only include numeric port, never full payload. |
| Backward compat with existing error consumers | New error shapes are additive; any legacy 500s for these cases are replaced — frontend must be audited (Open Question #1). |
| Multiple port columns in same row | `field` derived per-constraint; one mapped error at a time (pg raises one). |

**Unit — `portSchema`**

- Rejects: `0`, `65536`, `-1`, `3.5`, `"80"`, `NaN`, `undefined` (when not optional), `{}`, `true`.

**Unit — `createProjectSchema` / `patchProjectSchema`**
- Port fields optional + nullable.
- Invalid port propagates Zod error with correct path.
- `patchProjectSchema` allows partial; `createProjectSchema` requires required fields.

**Unit — `pgErrorMapper`**
- Table-driven for every known `<col>`: returns correct `{status, body}`.
- `23514` on non-matching constraint → rethrow.
- `23505` on non-matching constraint → rethrow.
- Unknown code (e.g. `23502`) → rethrow.
- Non-pg error (plain `Error`) → rethrow.
- `body.field` equals captured `<col>`; `body.value` equals payload port.

**Integration**
- `POST /api/projects` with port `70000` → 400 `invalid_port_range` (exercises real constraint).
- Duplicate port insert → 409 `port_conflict`.
- Unknown failure surfaces as 5xx.

**E2E:** not required (no UI changes); covered by downstream frontend story if/when wired.
