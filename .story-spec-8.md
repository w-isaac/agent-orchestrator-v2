# AOV-8: Prompt construction module with tiered context injection and token budget enforcement

As an agent dispatcher, I need a prompt construction module that assembles structured prompts using tiered context injection so that agents receive maximally relevant information within a strict token budget. The module must segment context artifacts by relevance score and inject them at appropriate fidelity levels: full content for high-relevance artifacts, concise summaries for medium-relevance, and one-line descriptions for low-relevance. The prompt template must include clearly delineated task, context, and constraints sections. The token budget must never be exceeded, with budget allocation calculated per the architecture document.


## Acceptance Criteria
- Given a set of context artifacts with relevance scores, when the prompt is constructed, then high-relevance artifacts are injected as full content, medium-relevance as summaries, and low-relevance as one-liners
- Given a configured token budget, when the prompt is assembled, then the total token count never exceeds the budget under any combination of inputs
- Given the prompt template, when rendered, then it contains distinct task, context, and constraints sections in the correct order
- Given the budget calculation logic defined in the architecture doc, when budget is computed, then the implementation matches that specification exactly
- Given unit tests for budget compliance, when the test suite runs, then all budget enforcement scenarios pass with zero violations
- Given an edge case where high-relevance content alone exceeds the budget, when the prompt is built, then content is truncated or downgraded gracefully without exceeding the limit

## Screen Descriptions

### 1. Prompt Builder View (primary screen)

**Layout:** Two-column split
- **Left column (40%):** Context Artifact Panel — lists all available artifacts with relevance scores and assigned fidelity tier
- **Right column (60%):** Prompt Preview Panel — live-rendered prompt showing Task / Context / Constraints sections

**Key UI elements:**
- Page header with title "Prompt Builder" + active token budget readout (e.g. `4,218 / 8,192 tokens`)
- Token Budget Meter — horizontal progress bar beneath header, color-coded (green → yellow → red)
- Artifact list with per-item fidelity badges (FULL / SUMMARY / ONE-LINE)
- Prompt preview with clearly labelled section dividers
- "Build Prompt" CTA button (disabled when budget is exceeded)

**States:**
| State | Behavior |
|---|---|
| Empty | Artifact list shows empty state illustration: "No context artifacts loaded." Prompt preview shows template skeleton with placeholder text in each section. |
| Loading | Artifact list shows skeleton cards. Budget meter shows indeterminate shimmer. |
| Populated | Artifacts rendered with relevance scores and auto-assigned tiers. Prompt preview updates in real time. |
| Budget exceeded | Budget meter turns red. Overflow indicator appears. "Build Prompt" button disabled with tooltip "Token budget exceeded." |
| Error | Banner at top of artifact list: "Failed to load artifacts. [Retry]" |

## ARCHITECTURE

Architecture spec is complete at `/app/architecture-spec.md`. Here's a summary of what it covers:

**Technical Approach** — Server-side prompt builder service with budget enforcement, tiered context injection (full/summary/one-line based on relevance scores), and real-time preview via WebSocket.

**Data Model** — 4 new tables: `context_artifacts`, `prompt_templates`, `prompt_builds`, `artifact_tier_overrides`. Migration `039_prompt_builder.sql`. All additive, no existing table alterations.

**API** — 6 new endpoints under `/api/prompt-builder/`:
- `GET /:storyId/artifacts` — artifacts with auto-assigned tiers
- `GET /:storyId/budget` — budget allocation breakdown
- `POST /:storyId/build` — assemble + persist prompt (422 if over budget)
- `POST /:storyId/preview` — assemble without persisting
- `PATCH /artifacts/:artifactId/override` — manual tier overrides
- `GET /templates` — list prompt templates

**File Changes** — 12 new files (migration, API router, service, token counter, 3 test files, 5 React components), 4 modified files (server.js, broadcaster, engineering-agent, client router).

**Budget Algorithm** — Waterfall allocation: Task 10%, Constraints 7%, Reserved 6%, Context ~77%. Artifacts processed in descending relevance order with automatic downgrade/truncation when budget is tight. 5% safety margin on all sections.

**Dependencies** — Only `gpt-tokenizer` (pure JS, no native deps).

**Risks** — 5 identified: token count accuracy (mitigated by 5% margin), truncation quality (sentence-boundary truncation), override abuse (server-side enforcement), migration safety (additive only), performance (pre-computed token counts).

**Testing** — 10 service unit tests, 4 token counter tests, 8 API integration tests, 6 component test suites, 1 E2E smoke flow.
