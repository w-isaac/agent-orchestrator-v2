# AOV-82: pipeline_stages table: DDL migration, default stage seeding, and project API inclusion

As a developer building the pipeline board, I need a pipeline_stages table in the database so that each project has a defined set of ordered stages with optional gate flags, and those stages are automatically available whenever a project is created or fetched. Without this foundation, no pipeline UI or stage-based story routing can function. The table must be seeded with 9 default stages on project creation, and the GET /api/projects/:id response must include the full ordered stage list. Stories will reference pipeline_stages via a current_stage_id foreign key.


## Acceptance Criteria
- Given the migration runs, the pipeline_stages table exists with columns: id (uuid PK), project_id (uuid FK → projects), name (text), icon (text nullable), stage_order (integer), has_gate (boolean default false), created_at (timestamptz)
- Given a new project is created via POST /api/projects, then 9 default stages are automatically inserted into pipeline_stages for that project in the correct order
- Given GET /api/projects/:id is called, then the response body includes a pipeline_stages array containing all stages for that project ordered by stage_order ascending
- Given stories have a current_stage_id column (FK to pipeline_stages.id), then the constraint is enforced at the DB level and NULL is allowed for stories not yet assigned a stage
- Given the migration is rolled back, then the pipeline_stages table and any FK columns added to stories are cleanly removed without data loss to other tables

## ARCHITECTURE

# Architecture Specification: `pipeline_stages` Table & Project API Inclusion

## Technical Approach

Backend-first foundation story. Add a `pipeline_stages` table with project FK, seed 9 default stages atomically inside the `POST /api/projects` transaction, and include the ordered stage list in `GET /api/projects/:id`. Add a nullable `current_stage_id` FK to `stories`. UI components from the UX spec are out-of-scope for this story but the API contract is shaped to support them.

Key principles:
- All seeding happens in a single DB transaction with project insert (atomic).
- Default stages defined as a single source-of-truth constant in code (not a separate seed migration) so they apply to every newly created project.
- Migration is reversible: down migration drops the FK column on `stories` first, then `pipeline_stages`.

## Data Model Changes

### New table: `pipeline_stages`

| Column | Type | Constraints |
|---|---|---|
| `id` | uuid | PK, default `gen_random_uuid()` |
| `project_id` | uuid | NOT NULL, FK → `projects(id)` ON DELETE CASCADE |
| `name` | text | NOT NULL |
| `icon` | text | NULL |
| `stage_order` | integer | NOT NULL |
| `has_gate` | boolean | NOT NULL DEFAULT false |
| `created_at` | timestamptz | NOT NULL DEFAULT `now()` |

**Indexes:**
- `idx_pipeline_stages_project_id` on `(project_id)`
- `uniq_pipeline_stages_project_order` UNIQUE on `(project_id, stage_order)` — prevents duplicate ordering
- `uniq_pipeline_stages_project_name` UNIQUE on `(project_id, name)` — prevents duplicate names per project

### Altered table: `stories`

| Change | Detail |
|---|---|
| Add column | `current_stage_id uuid NULL REFERENCES pipeline_stages(id) ON DELETE SET NULL` |
| Add index | `idx_stories_current_stage_id` on `(current_stage_id)` |

### Migrations

- `migrations/2026XXXXXXXXXX_create_pipeline_stages.{up,down}.sql`
  - **up:** create table, indexes, then `ALTER TABLE stories ADD COLUMN current_stage_id ...`, add stories index.
  - **down:** drop stories index, drop `stories.current_stage_id`, drop `pipeline_stages` indexes, drop table.

## API Changes

### `POST /api/projects` (modified)

- **Behavior change:** within the same transaction as project insert, insert 9 rows into `pipeline_stages` from the default-stages constant.
- **Request:** unchanged.
- **Response:** project body now includes `pipeline_stages: PipelineStage[]` (ordered by `stage_order ASC`).

### `GET /api/projects/:id` (modified)

- **Response shape:**
```json
{
  "id": "uuid",
  "name": "string",
  "...": "existing fields",
  "pipeline_stages": [
    {
      "id": "uuid",
      "project_id": "uuid",
      "name": "Backlog",
      "icon": "inbox",
      "stage_order": 1,
      "has_gate": false,
      "created_at": "2026-04-18T..."
    }
  ]
}
```

### `GET /api/projects` (modified, optional)

- Include `pipeline_stages` per project for parity. Recommended to keep response consistent.

### `POST /api/stories` and `PATCH /api/stories/:id` (modified)

- Accept optional `current_stage_id: uuid | null` in body.
- Validate FK belongs to story's project; return `400` on mismatch, `404` if stage not found.
