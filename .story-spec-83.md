# AOV-83: Stories table and full CRUD API

As a pipeline orchestrator, I need a stories database table and a complete CRUD REST API so that work units can be created, tracked, and progressed through pipeline stages. Stories are the core unit of work flowing through the agent pipeline — each represents a discrete task with a defined status lifecycle (queued → in_progress → gate → done/failed/cancelled). Without this foundation, no other pipeline stage or agent dispatch can function.


## Acceptance Criteria
- Given the application starts, when the stories migration runs, then the stories table exists with all required columns (id, project_id, title, description, acceptance_criteria, priority, epic, status, github_issue_number, created_at, updated_at) and a status enum covering: queued, in_progress, gate, done, failed, cancelled
- Given a valid POST /api/stories request with project_id and required fields, when the request is processed, then a new story is created with status defaulting to 'queued' and the created story is returned with HTTP 201
- Given a GET /api/stories request with an optional project_id query parameter, when the request is processed, then all matching stories are returned (filtered by project_id if provided) with HTTP 200
- Given a GET /api/stories/:id request for an existing story, when the request is processed, then the full story object is returned with HTTP 200; for a non-existent id, HTTP 404 is returned
- Given a PATCH /api/stories/:id request with valid fields, when the request is processed, then only the provided fields are updated, updated_at is refreshed, and the updated story is returned with HTTP 200
- Given a DELETE /api/stories/:id request for an existing story, when the request is processed, then the story is removed and HTTP 204 is returned; for a non-existent id, HTTP 404 is returned

## Technical Approach

Introduce the `stories` table as the foundational work-unit entity, owned by an Express router (`src/api/stories.js`) wired into the app at `/api/stories`. Persistence uses the existing `better-sqlite3` client with a forward-only SQL migration applied at boot. Status is enforced via a `CHECK` constraint covering the 6 enum values. The router exposes 5 REST endpoints (POST/GET-list/GET-by-id/PATCH/DELETE) returning JSON, with `project_id` validated against `projects(id)` via FK. `updated_at` is refreshed on every PATCH.

Client work delivers a Stories page driven by React Query hooks against the new endpoints, with 8 new components (table, row, badge, drawer, three modals, delete dialog) and a project filter that round-trips through the URL query string.

## Data Model Changes

### New table: `stories`

| Column | Type | Constraints |
|---|---|---|
| `id` | TEXT | PRIMARY KEY (UUID v4) |
| `project_id` | TEXT | NOT NULL, REFERENCES `projects(id)` |
| `title` | TEXT | NOT NULL, length ≤ 255 |
| `description` | TEXT | NULL |
| `acceptance_criteria` | TEXT | NULL (free text or JSON-encoded list) |
| `priority` | TEXT | NOT NULL DEFAULT `'medium'`, CHECK in (`low`,`medium`,`high`,`critical`) |
| `epic` | TEXT | NULL |
| `status` | TEXT | NOT NULL DEFAULT `'queued'`, CHECK in (`queued`,`in_progress`,`gate`,`done`,`failed`,`cancelled`) |
| `github_issue_number` | INTEGER | NULL |
| `created_at` | TEXT | NOT NULL (ISO-8601) |
| `updated_at` | TEXT | NOT NULL (ISO-8601) |

### Indexes
- `idx_stories_project_id` on `(project_id)` — supports the list filter.
- `idx_stories_status` on `(status)` — supports scheduler queries.
- `uniq_stories_project_github_issue` UNIQUE on `(project_id, github_issue_number)` WHERE `github_issue_number IS NOT NULL` — prevents duplicate GitHub issue mapping per project.

### Migration
- File: `src/db/migrations/0XX_stories.sql` (next sequential number).
- Forward-only `CREATE TABLE IF NOT EXISTS` + index DDL. Applied automatically by the existing migration runner on server start.

## API Changes

Base path: `/api/stories`. All bodies/responses are JSON. Errors use `{ error: string }`.

| Method | Path | Request body | Success | Errors |
|---|---|---|---|---|
| `POST` | `/api/stories` | `{ project_id*, title*, description?, acceptance_criteria?, priority?, epic?, github_issue_number? }` | `201` + full story | `400` missing required, `404` unknown project, `409` duplicate GitHub issue |
| `GET` | `/api/stories?project_id=&status=` | — | `200` `Story[]` (sorted `created_at DESC`) | — |
| `GET` | `/api/stories/:id` | — | `200` full story | `404` not found |
| `PATCH` | `/api/stories/:id` | Any subset of `{ title, description, acceptance_criteria, priority, epic, status, github_issue_number }` | `200` updated story | `400` no/invalid fields, `404` not found |
| `DELETE` | `/api/stories/:id` | — | `204` no content | `404` not found |

**Response shape (story):**
```json
{
  "id": "uuid", "project_id": "uuid", "title": "...", "description": "...",
  "acceptance_criteria": "...", "priority": "medium", "epic": "...",
  "status": "queued", "github_issue_number": 42,
  "created_at": "2026-04-18T...", "updated_at": "2026-04-18T..."
}
```

**PATCH semantics:** only fields present in the body are updated; `status` value is validated against the enum; `updated_at` is set server-side.
