# AOV-84: Story lifecycle: advance/retreat stages, gate approval, dependency management, and prioritization

As a pipeline operator, I need full lifecycle control over stories so that I can move them forward and backward through pipeline stages, approve gated transitions, manage inter-story dependencies with safety guardrails, and reorder stories within a stage to reflect changing priorities. Without these controls, agents cannot be dispatched in the correct order and the pipeline cannot enforce stage gates or dependency contracts.


## Acceptance Criteria
- Given a story in any non-terminal stage, when POST /api/stories/:id/advance is called, then the story moves to the next pipeline stage and the updated story is returned with the new stage
- Given a story in any non-initial stage, when POST /api/stories/:id/retreat is called, then the story moves to the previous pipeline stage and the updated story is returned with the new stage
- Given a stage that requires gate approval, when POST /api/stories/:id/approve is called with valid approver context, then the gate is marked approved and the story may advance
- Given a story with an unapproved gate, when advance is attempted without approval, then the API returns 422 with a descriptive error indicating gate approval is required
- Given a valid dependency target, when PUT /api/stories/:id/dependencies is called, then the dependency is persisted and returned in the story's dependency list
- Given a dependency addition that would create a circular dependency chain, when PUT /api/stories/:id/dependencies is called, then the API returns 422 with a 'circular dependency detected' error and no change is made
- Given stories in the same stage, when POST /api/stories/:id/prioritize is called with direction 'up' or 'down', then the story's position within the stage is updated relative to its neighbors
- Given any lifecycle mutation endpoint, when the operation succeeds, then a WebSocket broadcast is emitted with the updated story payload so connected clients receive real-time updates

# Architecture Specification: Story Lifecycle Management

## Technical Approach

Extend existing story service with lifecycle controller handling advance/retreat/approve/dependencies/prioritize. Introduce a stage registry (ordered list + gate flags), a `story_gates` table for approval state, a `story_dependencies` edge table with DFS cycle detection, and a `position` column for intra-stage ordering. All mutations emit WebSocket broadcasts via an existing `wsBroadcaster` (or new `ws/stories` channel) after DB commit. Transactional boundaries wrap each lifecycle mutation to keep stage/gate/position consistent. React side subscribes to the channel, applies optimistic updates, and reconciles on broadcast.

## Data Model Changes

### New tables

**`story_dependencies`**
| column | type | notes |
|---|---|---|
| `story_id` | uuid | FK stories(id), on delete cascade |
| `depends_on_id` | uuid | FK stories(id), on delete cascade |
| `created_at` | timestamptz | default now() |

- PK `(story_id, depends_on_id)`
- Index `idx_deps_depends_on` on `depends_on_id` (reverse lookup for cycle detection)
- CHECK `story_id <> depends_on_id`

**`story_gates`**
| column | type | notes |
|---|---|---|
| `id` | uuid | PK |
| `story_id` | uuid | FK stories(id) |
| `stage` | text | stage the gate guards |
| `approved` | boolean | default false |
| `approver_name` | text | nullable |
| `approval_reason` | text | nullable |
| `approved_at` | timestamptz | nullable |

- Unique `(story_id, stage)`
- Index `(story_id)`

**`story_history`**
| column | type | notes |
|---|---|---|
| `id` | uuid | PK |
| `story_id` | uuid | FK stories(id) |
| `event_type` | text | `advance` \| `retreat` \| `approve` \| `dep_add` \| `dep_remove` \| `prioritize` |
| `from_value` | jsonb | stage / position / dep before |
| `to_value` | jsonb | after |
| `actor` | text | nullable |
| `created_at` | timestamptz | default now() |

- Index `(story_id, created_at DESC)`

### Altered tables

**`stories`**
- Add `position` integer NOT NULL DEFAULT 0 — ordering within a stage
- Add `stage` text NOT NULL (if not already present)
- Index `(stage, position)` for ordered reads

## API Changes

| Method | Path | Request | Response |
|---|---|---|---|
| POST | `/api/stories/:id/advance` | `{}` | `200 { story }` \| `422 { error: "gate_approval_required" }` \| `409` if at terminal stage |
| POST | `/api/stories/:id/retreat` | `{}` | `200 { story }` \| `409` if at initial stage |
| POST | `/api/stories/:id/approve` | `{ approver_name: string, approval_reason?: string }` | `200 { story, gate }` \| `400` invalid |

| GET | `/api/stories/:id/history` | — | `200 { events: [...] }` |
| GET | `/api/stories/:id/dependencies` | — | `200 { dependencies: [...], dependents: [...] }` |

**WebSocket broadcast** — on each successful mutation:
```
{ type: "story.updated", payload: { story, cause: "advance"|"retreat"|"approve"|"deps"|"prioritize" } }
```
